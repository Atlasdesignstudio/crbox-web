## Core product principle
This calculator must not imply false precision.

Users often do not know:
- the exact final package dimensions
- the exact real shipping weight
- the exact packaged form factor
- the exact final customs/tariff reality

So the calculator must feel like:
- a smart estimate tool
- useful for orientation and comparison
- clearly framed as an estimate based on user-entered values
- with a real path to send the estimate to CRBOX for confirmation

This must be reflected in UI, copy, and handoff flow.

## What the send flow should include
The sent quote must be intelligent and concise, including the minimum relevant information needed for CRBOX to quote properly.

That means the generated email / WhatsApp message should include:

### User contact
- user email
- optional user name if already collected / available

### Quote context
- whether this was viewed as:
  - consolidated
  - separate
- destination selected
- estimated total shown
- whether the result is single-item or multi-item
- savings amount if a consolidation comparison exists

### Article-level details
For each item, include:
- product name
- product URL, if provided
- declared value
- entered real weight
- entered dimensions
- volumetric weight shown by the calculator
- selected / suggested category
- tariff source label if relevant

### Important framing
The generated message should clearly state that:
- this is an estimate based on user-entered values
- final cost may vary depending on the actual package received by CRBOX
- the user is requesting confirmation / review from CRBOX

Do not overload the message with unnecessary UI text or long explanations.
It should feel useful, readable, and operational.

## Main problems to solve

### 1. Invisible / white CTAs
Several CTAs clearly exist and are clickable, but visually render as white / blank / nearly invisible.

This is happening in multiple places, including:
- the top-right CTA inside “Tus artículos”
- the primary CTA inside the “Agregar artículo” modal
- the CTA below “Destino de entrega” for calculating / comparing shipments
- some tab-like controls that visually look disabled/transparent in the results area

This must be fixed fully and consistently.

#### Requirement
- every interactive CTA must have a clear visible state
- primary, secondary, disabled, active, and inactive states must be unmistakable
- no ghost button should appear visually broken if it is clickable
- if something is disabled, it must look intentionally disabled, not broken

---

### 2. “Agregar artículo” modal needs a better product UX
The add-item modal currently feels mechanically functional but not fully useful.

#### Specific issues
- the URL field currently does not create enough value for the user
- “Inferido” / “confirmar” is not clear enough
- the inferred-state chips are visually awkward
- the USD symbol overlaps visually inside declared value
- spacing at the top feels cramped / too close
- the primary CTA in the modal is visually broken

#### Product direction
The URL input should become useful in a downstream way.

Even if inference is limited, the entered product URL should be stored as part of the quote context so that later, when the user sends the estimate to CRBOX, the team can inspect the exact product link.

So:
- URL should not be treated as decorative
- it should become part of the eventual quote payload / summary sent to CRBOX
- if no useful metadata is extracted, the URL still has value as reference context

#### What to improve
- make the URL field feel clearly optional but useful
- clarify what is inferred and what still requires manual user confirmation
- redesign the inferred-state treatment so it feels elegant and legible
- fix the value field composition so currency symbol, amount, and inferred badge do not visually collide
- improve modal spacing, field grouping, and action hierarchy
- fix the modal primary CTA visual rendering completely

- CRBOX may confirm or adjust based on the actual received package

If you propose a better solution than a simple “margin”, use it — but the core problem must be solved:

This is one of the most important additions.

#### Important
The user should not have to write any custom message.
