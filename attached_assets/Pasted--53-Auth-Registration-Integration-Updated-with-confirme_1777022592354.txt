CONFIRMED LOGIN CONTRACT
==================================================

Endpoint:
POST https://clients.crbox.cr/authtoken

Content-Type:
application/x-www-form-urlencoded

Body shape:
grant_type=password&username={email}&password={password}

Confirmed successful response fields:
- access_token
- token_type
- expires_in

Confirmed usage:
- Update Profile uses this token as:
  Authorization: Bearer {token}

This means login is a Bearer-token auth flow and can be implemented now.

Still unknown:
- failed login response format
- whether a test-environment login endpoint exists
- whether direct browser calls will hit CORS from the actual frontend origin

CONFIRMED CREATE CLIENT / REGISTRATION CONTRACT
==================================================

Endpoint:
POST https://test.clients.crbox.cr/api/crboxwebapi/postregisteruser

Content-Type:
application/x-www-form-urlencoded

Important request pattern:
- request body is application/x-www-form-urlencoded
- Phones and Addresses are JSON-stringified arrays embedded as string values inside the form body
- do not send a JSON body

Confirmed response behavior:
- API returns HTTP 200 for both success and observed errors
- frontend must determine outcome from JSON body, not HTTP status

Confirmed response envelope:
{
  "StatusResult": "OK" | "ERROR",
  "Message": "..."
}

Confirmed observed Create Client responses:
- success:
  { "StatusResult": "OK", "Message": "" }

- duplicate identification:
  { "StatusResult": "ERROR", "Message": "El número de identificación ya está registrado en la base de datos" }

- duplicate email:
  { "StatusResult": "ERROR", "Message": "El correo ya existe en la base de datos" }

Confirmed successful Create Client field set includes:
Identity/profile fields:
- Consignee.ConsigneeName
- Consignee.ConsigneeLastName1
- Consignee.ConsigneeLastName2
- Consignee.IdentificationNumber
- Consignee.IdentificationType
- Consignee.IsCompany

- Consignee.ResidenceCountry
- Consignee.ReceivesNewsletter
- Consignee.Responsabilidad
- Consignee.IdResponsabilidad
- Consignee.ContactName1
- Consignee.ContactName2
- Consignee.AlternativeEmail
- CompanyCode

Email/password fields:
- Consignee.Email
- ConfirmEmail
- Password
- ConfirmPassword

Branch/sucursal field:
- Consignee.Sucursal.IdSucursal

Complex fields:
- Phones = JSON-stringified array
- Addresses = JSON-stringified array

Confirmed Create Client implementation rules:
- frontend must always inspect StatusResult
- frontend must not treat HTTP 200 as success by itself
- known backend business errors already confirmed:
  - duplicate email
  - duplicate ID
