Context:
We already confirmed the correct upload compatibility behavior:

- `FileLocation` must be the full absolute file URL returned by saveBill, for example:
  `https://crbox.cr/wp-content/uploads/...`
  or the equivalent WordPress media URL returned by the upload proxy.

- `ClientInvoiceText` must be clean sanitized description/text HTML only, for example:
  `<p>description text</p>`

- `ClientInvoiceText` must NOT embed:
  - `<img src="...">`
  - `<a href="...">`
  - `https://crbox.cr`
  - `https://wp.crbox.cr`
  - `/wp-content/uploads`
  - `/uploads/invoices`

The file itself belongs in `FileLocation`, not inside `ClientInvoiceText`.

Known client-facing endpoints from the old portal:

1. Get purchase bills:
`GET https://clients.crbox.cr/api/crboxwebapi/getpurchasebills/{warehouseReceiptId}`

Observed response includes:
- `IdPurchaseBill`
- `FileLocation`
- `FileTextContent`
- `NumeroFactura`
- `Monto`
- `Descripcion`
- `CreatedDate`

2. Delete purchase bill:
`GET https://clients.crbox.cr/api/crboxwebapi/getdeletepurchasebill/{warehouseReceiptId}/{email}`

Observed response:
`{"StatusResult":"OK","Message":"ok se borro"}`

Request uses:
- `Authorization: Bearer <user token>`
- logged-in user email

3. Create purchase bill:
`POST https://clients.crbox.cr/api/crboxwebapi/postcreatepurchasebill`

Expected final payload:
- `WRId`
- `NumeroFactura`
- `Monto`
- `Descripcion`
- `ClientInvoiceText = <p>{sanitized description/text}</p>`
- `FileLocation = {full absolute file URL returned by saveBill}`

Scope approved:

A. Add API support in the new portal

Add or confirm wrappers for:

1. `getPurchaseBills(warehouseReceiptId)`
- Calls `GET /api/crboxwebapi/getpurchasebills/{warehouseReceiptId}` or the existing direct legacy equivalent used by the portal.
- Sends the logged-in user Bearer token.
- Returns normalized bill data:
  - idPurchaseBill
  - fileLocation
  - fileTextContent
  - numeroFactura
  - monto
  - descripcion
  - createdDate

2. `deletePurchaseBill(warehouseReceiptId, email)`
- Calls:
  `GET getdeletepurchasebill/{warehouseReceiptId}/{email}`
- Sends Authorization Bearer token.
- Requires `StatusResult === "OK"` to count as success.
- Does not expose token, email, PII, file URL, invoice number, or WR details in logs.

B. Add “Eliminar factura” UI

Show delete action only when a factura exists for the package.

Before deleting, show confirmation:

“¿Eliminar factura?
Esta acción eliminará la factura asociada a este paquete. Podrás subir una nueva después.”

On confirm:
1. Call `deletePurchaseBill(warehouseReceiptId, loggedInEmail)`.
2. If success:
   - clear local invoice URL cache for that WR
   - clear any `data-invoice-url` or equivalent attribute
   - update invoicesCount / local bill state
   - refresh the package row or reload package list
   - show success toast:
     “Factura eliminada correctamente.”
   - UI should return to “Subir factura”
3. If failure:
   - do not change UI state
   - show clear error toast:
     “No pudimos eliminar la factura. Inténtalo de nuevo.”

C. Add “Reemplazar factura” UI

Do not implement true in-place editing.

“Reemplazar factura” means:
delete existing factura, then upload a new one using the existing corrected upload flow.

Confirmation copy:

“Reemplazar factura
Primero eliminaremos la factura actual y luego podrás subir una nueva. Si la nueva carga falla, tendrás que intentarlo nuevamente.”

Flow:
1. User clicks “Reemplazar factura”.
2. Show confirmation.
3. If user confirms, call `deletePurchaseBill`.
4. If delete succeeds:
   - clear local invoice cache/state for that WR
   - open the existing upload factura modal
   - preselect / preserve the same WR package context
   - user uploads new factura using the existing upload flow
5. Existing upload flow must send:
   - `FileLocation = full absolute file URL`
   - `ClientInvoiceText = sanitized description/text only`
6. After successful upload:
   - refresh package row / package list
   - show success toast:
     “Factura reemplazada correctamente.”
7. If delete succeeds but upload fails:
   - show recovery message:
     “La factura anterior fue eliminada, pero la nueva no se pudo subir. Por favor intenta subirla nuevamente.”
   - keep the package in a clear state where user can retry upload.
8. If delete fails:
   - do not open upload modal
   - do not alter current UI state
   - show error toast.
