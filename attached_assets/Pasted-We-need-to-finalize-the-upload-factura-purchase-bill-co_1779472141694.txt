D. Initial new portal bug found

The new portal upload flow previously did this:

1. Upload file through current upload path.
2. Receive a full URL, for example:

https://crbox.cr/uploads/invoices/9fcf7110-6f58-4203-acd2-50f60bbb7852.png

or WP-style:

https://crbox.cr/wp-content/uploads/...

3. Build `ClientInvoiceText` with embedded full URL:

<img src="https://crbox.cr/uploads/invoices/..." />

or PDF link:

<a href="https://crbox.cr/wp-content/uploads/...">...</a>

4. Separately compute `_filePath` after building `ClientInvoiceText`.

Previous code pattern:

var _safeUrl  = fileUrl ? fileUrl.replace(/"/g,'&quot;') : '';
// ClientInvoiceText built here using _safeUrl
var _filePath = (function(u) {
  if (!u) return '';
  try { return new URL(u).pathname; } catch (_) { return u; }
}(fileUrl));

This meant `ClientInvoiceText` used full absolute URLs.

A path-normalization fix was implemented:

var _filePath = (function(u) {
  if (!u) return '';
  try { return new URL(u).pathname; } catch (_) { return u; }
}(fileUrl));

var _safeUrl = _filePath ? _filePath.replace(/"/g,'&quot;') : '';

After that:
- `_filePath` is computed before `_safeUrl`
- `_safeUrl` is derived from `_filePath`
- `ClientInvoiceText` uses a pathname instead of a full URL
- `FileLocation` still uses `_filePath`
- `_invoiceUrlCache` still uses the full `fileUrl` for client display

However, additional old portal evidence now refines this conclusion.

F. Important correction to previous assumption

The old portal’s working behavior proves:

FileLocation = full absolute WordPress URL

is valid and expected.

Specifically:

FileLocation = https://wp.crbox.cr/wp-content/uploads/2026/05/FacturaCompra-WR606979.pdf

Therefore, the root problem is probably NOT full absolute URL in `FileLocation`.

The stronger root cause is:

The new portal was embedding the invoice file URL inside ClientInvoiceText, while the old portal does not.

Old portal working convention:

FileLocation = full absolute WP URL
FileTextContent / ClientInvoiceText = clean text/HTML only, no embedded file URL

New portal broken convention:

FileLocation = maybe OK or normalized
ClientInvoiceText = embedded full image/PDF URL

This likely caused double-host rendering and/or stored malformed HTML.

Therefore, the safest compatibility approach may be:

FileLocation should match old portal behavior:
https://wp.crbox.cr/wp-content/uploads/...

ClientInvoiceText should be clean description/text HTML only:
<p>...</p>

Do NOT embed <img src="..."> or <a href="..."> file links inside ClientInvoiceText unless absolutely necessary and proven safe.

Please verify whether the recently implemented path-normalization fix is sufficient or whether a small final correction is needed to match the old portal exactly.
