# Task — Domain Cutover & Rollback Readiness Plan

What & Why
We are preparing the CRBOX new website/portal to eventually become the primary production experience. Before changing any domain/DNS, we need a detailed cutover and rollback plan that maps what currently depends on the old site, what the new site already handles, what still goes through legacy APIs, and how to switch safely without downtime.

This is planning only. Do not change DNS, domains, hosting, production flags, AWS, Replit deployment, or legacy systems.

Goal
Create a complete domain cutover readiness document for moving public and portal traffic safely to the new deployment when the system is ready.

Deliverable
Create:
docs/domain-cutover-readiness-plan.md

Scope
Audit and document:
- current public domain behavior
- current legacy WordPress/site behavior
- new Replit/Vercel deployment behavior if applicable
- portal page routes
- login route
- signup/afiliación route
- password recovery
- public pages
- redirects
- canonical URLs
- sitemap/robots
- GA4/GTM
- Search Console
- SSL/TLS
- rollback plan

Do not modify anything in this task.

## 7. Cutover stages

Define stages:

### Stage 0 — Current state
Old domain/site active. New site/portal under development or staging. Legacy APIs remain the source for auth/writes.

### Stage 1 — Staging validation
New deployment tested on staging URL. No DNS change. Internal QA only.

### Stage 2 — Soft launch
Limited internal testing with real flows. No DNS change. Production APIs may be used only if already approved.

### Stage 3 — DNS cutover
Domain points to new deployment. Legacy system remains available behind the scenes.

### Stage 4 — Post-cutover monitoring
Monitor:
- logs
- analytics
- forms
- login
- signup
- password recovery
- packages
- invoices
- mi-cuenta
- mobile navigation
- SEO/indexing
- error rates

### Stage 5 — Legacy decommission planning
Only after a confidence period. Do not decommission legacy until all dependencies are replaced or intentionally retained.

## 9. Rollback plan

Document exact rollback:
- DNS record revert
- expected TTL impact
- what to check after rollback
- keeping legacy system untouched
- preserving old hosting until stable
- who is responsible for rollback
- how to communicate rollback internally
- what logs/metrics indicate rollback is needed

Rollback triggers:
- login broken
- signup broken without workaround
- password recovery broken without workaround
- portal unavailable
- forms unavailable
- high 5xx error rate
- analytics not firing on critical flows
- incorrect redirects
- private portal pages indexable
- severe mobile navigation issue
- any security-sensitive data exposed

Rollback checklist:
- revert DNS to previous target
- confirm old site loads
- confirm login works on old site
- confirm public pages load
- confirm SSL valid
- confirm GA4/GTM not duplicated in a harmful way
- monitor for 24 hours

## 10. Go / No-Go checklist

No-Go criteria:

- forms not working

- public routes missing or broken
- mobile navigation broken

## 12. Legacy preservation and decommission policy

Document:
- legacy site must remain available after cutover
- legacy APIs remain required for auth/writes
- old hosting should not be deleted immediately
- legacy decommission should be a separate future project
- dependencies must be retired one by one

Confidence period recommendation:
- minimum 1–2 weeks after cutover before considering any legacy reduction
- longer if signup/auth/writes still depend on legacy
