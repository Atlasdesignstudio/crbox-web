How I would write the plan:

#243 - Invoice Upload Flow — Mis Paquetes + Mis Facturas

## What & Why
The new portal already shows real package data in Mis Paquetes, but the invoice flow is still incomplete. Users can see packages, but they cannot yet upload the purchase invoice directly from the new experience in the same practical way they can in the live/legacy portal.

We already verified the real production backend flow and the exact identifiers involved, so this task should not invent anything new. It should simply connect the new portal UI to the real existing legacy contracts and make the experience usable across desktop and mobile.

This task completes the feature in a fully operational way by doing four things together:

1. adds a real Subir factura action per package in both table view and card view  
2. adds a clear Pendientes de factura filter so users can quickly find packages that still need an invoice  
3. wires the real two-step legacy upload flow (saveBill → postcreatepurchasebill) using the correct package identifier  
4. ensures uploaded invoices are reflected back in Mis Paquetes and surfaced truthfully in Mis Facturas

This is not a redesign and not a backend replacement. It is a real integration + UX completion task.

---

## Verified backend contracts

### Step 1 — upload invoice file
POST https://crbox.cr/wp-json/crbox/v1/saveBill

Content-Type: multipart/form-data

Fields
- email
- invoice (binary file)
- wr_id

Verified response
- file
- url
- type

This step uploads the file and returns the final uploaded file URL.

### Step 2 — create purchase bill record
POST https://clients.crbox.cr/api/crboxwebapi/postcreatepurchasebill

Content-Type: application/x-www-form-urlencoded

Fields
- ClientInvoiceText
- Descripcion
- FileLocation
- Monto
- NumeroFactura
- WRId

Verified success response
- StatusResult = "OK"
- Message = "La factura fue almacenada satisfactoriamente"

This step creates the actual invoice/bill record in the CRBOX system.

### Step 3 — refresh packages
After success, the live site refreshes the package list through:
GET /api/crboxwebapi/getuserpackages/...

The refreshed package response then reflects invoice state through real backend fields like:
- idwarehousereceipt
- invoicesCount
- montofactura
- descripcionfactura
- impresoFactura

The new portal must follow this same truth model.  
It should not rely on optimistic local-only state as the final source of truth.

## Critical identifier
The correct package identifier is:

- idwarehousereceipt

This exact field must be used as:
- wr_id in saveBill
- WRId in postcreatepurchasebill

Do not use tracking number as the linkage key.  
Do not derive another ID when idwarehousereceipt already exists.

---

## Deterministic technical rules
These rules are part of the implementation and should not be guessed:

1. Invoice presence source of truth
   - Use invoicesCount > 0 as the primary source of truth for whether a package already has an invoice.
   - Secondary fields like montofactura or descripcionfactura may be used only as fallback/debug checks if needed, not as the main state rule.

2. ClientInvoiceText format
   - Use the same simple legacy-compatible format:
   - ClientInvoiceText = <p>${description}</p>

3. Real refresh after success
   - After successful upload + bill creation, refetch packages through the existing package loader for the current filter/date state.
   - Do not rely on local state mutation as final truth.
   - Do not hard reload the full page unless there is no cleaner option.

4. Mis Facturas truthfulness
   - Do not fabricate package linkage in Mis Facturas.
   - Only show linked receipt/package/tracking/status if current real backend responses actually expose that information.
   - If the invoices endpoint does not expose enough linkage, show only what is real.

5. File validation
   - Allow the same practical file types as the current real flow: PDF and common image formats.
   - Validate before submit.
   - Show clear error messages if invalid.

## Done looks like

### Mis Paquetes — both layouts support invoice upload
In table view and card view, each package clearly shows one of these actions:

- Subir factura → if invoicesCount == 0
- Ver factura → if invoicesCount > 0

This must work in:
- desktop table view
- desktop card/grid view
- mobile

The invoice action must not exist only in one layout.

### Mis Paquetes — pending invoice visibility
Users can quickly find packages that still need invoice upload.

A clear filter or chip exists, such as:
- Pendientes de factura

This filter works using real backend state:
- invoicesCount == 0

It must work consistently in both:
- table view
- card view

### Upload modal/form
Clicking Subir factura opens a polished, responsive modal/form that collects:

- invoice number
- amount in USD
- description
- file upload

The form feels premium and easy to use, but remains fully compliant with the backend’s real requirements.

### Real two-step backend flow
Submitting the form runs the exact real flow in this order:

1. call saveBill
2. then call postcreatepurchasebill
3. then refetch packages

No fake upload state.
No shortcut flow.
No alternate backend.

### Package row/card updates from real backend data
After success, the portal refetches package data and rerenders invoice state from the refreshed real response.  
That means the package moves from Subir factura to Ver factura only when confirmed by the backend refresh.

### Mis Facturas reflects uploaded invoices
Uploaded invoices appear in Mis Facturas through the real backend flow.

Where truthfully available, the UI shows:
- invoice number
- amount
- linked receipt/package number
- tracking number
- package state/status

If some of that data is not truly available from the invoices backend, it must not be fabricated.

## Success and failure handling

### Success
If both backend calls succeed:
- refetch packages
- rerender package state from fresh backend data
- update invoice action accordingly
- allow invoice to be discoverable in Mis Facturas

### Failure A — saveBill fails
If file upload fails:
- do not call postcreatepurchasebill
- show clear error
- do not mark package as invoiced
- keep the user in a recoverable state

### Failure B — postcreatepurchasebill fails after upload
If upload succeeds but bill creation fails:
- do not mark the package as invoiced
- show clear error
- do not show fake success
- log enough context to debug

### Failure C — backend write succeeds but package refresh fails
If both backend steps succeed but package refresh fails:
- show a partial-success style message
- do not invent final invoice state
- allow retry / refresh path
- keep the UI honest
