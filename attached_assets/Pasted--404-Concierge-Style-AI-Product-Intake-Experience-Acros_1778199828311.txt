#404 - Concierge-Style AI Product Intake Experience Across Public + Portal Quote Flow

What & Why

CRBOX already has the Product Brain and AI-first classifier foundation. However, the visible quote experience still feels too technical, form-heavy, and dropdown-first.

The goal of this task is to redesign the product intake experience into a more human, concierge-style, AI-native flow across:

- Public Cotizar page
- Client portal quote/cotización flow
- Multi-product quote flow
- Product URL interpretation
- Calculadora handoff
- Floating CRBOX AI assistant
- Admin/sales structured quote context

The user should not feel like they are filling out customs software.

The user should feel like:

“Tell CRBOX what you want to bring, and we’ll help you understand taxes, restrictions, and next steps.”

This task is UX + interaction + AI orchestration.

Do not rewrite calculator math.
Do not break existing quote submission.
Do not break admin quote logic.
Do not break legacy category codes.
Do not remove existing functionality.

---

## Core Experience Shift

Move from:

- form-first
- dropdown-first
- technical category selection
- fragile URL crawling
- hard validation errors

To:

- AI concierge intake
- natural product description
- optional link
- optional price/value
- product understanding
- tax guidance
- restriction guidance
- progressive next steps

The first interaction should not be:

“Choose category.”

It should be:

“¿Qué te gustaría traer?”

## Product Price / Value Input

The user should be able to optionally provide the approximate product price early in the flow.

The price field is useful because it helps:

- give more accurate tax guidance
- improve Gemini/Product Brain context
- improve admin quote context
- improve estimate ranges
- reduce uncertainty

Important:

The price field is optional.

Do not block the experience if the user does not provide it.

Do not make it feel like a mandatory customs value field at the first step.

This field is for guidance, not final liquidation.

---

Gemini + Product Brain should use:

- product description
- optional URL
- approximate product value
- category
- risk flags

to provide more useful guidance.

---

## If Price Is Not Provided

The UX should still feel complete and successful.

Gemini should still provide:

- category understanding
- approximate tax guidance
- compliance guidance
- next-step guidance

using category-level ranges only.

Example:

“Los zapatos normalmente pagan alrededor de 29% en impuestos, aunque el monto final depende del valor, materiales y revisión final.”

Do not show:

“Price missing error.”

Instead show:

“Puedes agregar el precio para recibir una orientación más precisa.”

---

## If Gemini / URL Can Infer Price

If the URL provides a detectable price:

- prefill the price field softly
- mark as “Precio detectado automáticamente”
- allow the user to edit it

Never block the experience if price extraction fails.

---

## URL Is Context, Not a Scraping Mission

The current crawling/scraping approach is too fragile and creates broken-feeling UX.

Do not optimize the experience around perfect scraping.

The URL should be treated as contextual assistance.

The system may attempt to extract:

1. product title/name
2. category hints
3. price if available

But the experience should not depend on extracting:

- exact dimensions
- exact weight
- perfect title
- perfect price
- complete specs

If extraction is incomplete, the UX should still feel successful.

Examples:

“Ya entendimos el tipo de producto.”

“Detectamos una categoría probable.”

“Solo falta confirmar algunos detalles.”

Avoid:

- “No se pudo analizar”
- “AI failed”
- “No match”
- “Extraction error”
- “Producto no encontrado”

---

## AI Interpretation Flow

The ideal flow:

User input  
→ optional URL  
→ optional price  
→ Gemini interpretation  
→ Product Brain validation  
→ semantic approximation if needed  
→ friendly concierge explanation  
→ next-step routing  
→ existing calculator/admin systems

Gemini interprets.

Product Brain decides.

Calculator calculates.

Admin reviews.

## Product Brain + Gemini Rules

Use the existing:

- Product Brain
- CRBOXProductClassifier
- Gemini infrastructure
- legacy category mapping
- existing calculator engine
- existing quote payloads
- existing admin flow

Gemini may:

- interpret natural language
- infer product type
- use the URL as context
- use price as context
- semantically approximate to closest category
- detect risk patterns

Gemini must not:

- invent tax rates
- invent Product Brain categories
- calculate final shipping totals
- make final legal/customs decisions
- bypass Product Brain
- expose raw reasoning to customers

Product Brain must provide:

- category
- legacy category code
- estimated tax range
- totalEstimatedRate
- customerMessage
- manualReviewRequired
- automaticEstimateAllowed
- regulatedProduct
- restrictedProduct
- forbiddenProduct
- riskFlags
- adminNotes
- actionForCustomer
- actionForAdmin

“Continuar a calculadora”

- product description
- interpreted product name
- optional URL
- optional price/value
- Product Brain category
- legacy category code
- customerMessage
- manualReviewRequired status

If the product is automaticEstimateAllowed:

- allow normal calculator estimate

- show review message
- do not show misleading final-looking automatic total

## Floating CRBOX AI Assistant

The floating CRBOX AI assistant must become part of this same product intake ecosystem.

It should not feel like a separate generic chatbot.

It should behave like the same concierge.

The assistant should use:

- CRBOXProductClassifier
- Product Brain
- same Gemini prompt
- same customerMessage
- same semantic approximation
- same risk handling
- same next-step routing

Examples:

User:

“Quiero traer unos AirPods”

Assistant:

“Los AirPods normalmente pertenecen a Electrónica y suelen pagar aproximadamente 14–15% en impuestos.”

Then:

“¿Quieres recibir una estimación de envío o prefieres solicitar una cotización con CRBOX?”

---

User:

“Quiero traer proteína”

Assistant:

“Los suplementos pueden requerir validación sanitaria antes de importarse. CRBOX puede ayudarte a revisarlo antes de que hagas la compra.”

---

User:

“Quiero traer esto: [URL]”

Assistant:

“Ya entendimos el tipo de producto. Parece pertenecer a la categoría Calzado. ¿Quieres calcular un estimado o enviar una solicitud a CRBOX?”

---

User:

“Cuesta como $180”

## Calculadora Relationship

The calculadora should remain available for users who want an estimate.

But the intake experience should make it easier to reach calculadora with prefilled context.

Calculadora should not be the first required step.

It is one possible next step.

The concierge card should route users toward calculadora when:

- product is common
- automaticEstimateAllowed is true
- price is available or user wants estimate

Route toward CRBOX quote/review when:

- manualReviewRequired is true
- regulatedProduct is true
- forbidden/restrictedProduct is true
- product is ambiguous
- user wants precise quote
- product is high-value

## No Hard Failure UX

Normal uncertainty should never feel like failure.

If the exact product does not exist in the Product Brain:

Use semantic approximation.

If still unclear:

Use helpful guidance.

Example:

“Detectamos una categoría probable, pero CRBOX puede revisarla antes de confirmar.”

If the URL provides little data:

“No pudimos leer todos los datos del enlace, pero puedes completar lo que falta manualmente.”

If price is missing:

“Puedes agregar el precio para recibir una orientación más precisa.”
