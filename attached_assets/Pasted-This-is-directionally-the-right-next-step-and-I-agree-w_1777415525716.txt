This is directionally the right next step, and I agree with several parts of the proposed task:

## What I agree with
1. The portal should stop showing only “Revisa tu correo” once response_json exists.
2. The reviewed CRBOX response should be rendered natively inside solicitud.html.
3. The active solicitudes badge counts in mis-solicitudes.html and dashboard.html are valid and should be populated.
4. A dedicated portal-auth endpoint is the right direction rather than broadening the admin-only status endpoint.

However, there is an important product and state-model refinement needed before proceeding, because the current proposed customer action model is too simple.

--------------------------------------------------
Business reality that must be reflected
--------------------------------------------------

After CRBOX sends a reviewed response, the customer does not have only one generic next action.

There are actually two different customer intents:

1. Comprar con CRBOX  
CRBOX purchases the item on behalf of the customer and charges a 5% service fee with a $5 minimum.

2. Lo compraré por mi cuenta  
The customer purchases the item themselves, but still wants the system to track the expected arrival of that package to the CRBOX Miami locker / casillero flow.

This distinction is critical and should change the portal-side UX and state model.

--------------------------------------------------
Required refinement to the proposed task
--------------------------------------------------

Please do not model the next step as a single generic “Confirmar mi compra” action that transitions directly from respondida to completada.

That is too semantically strong and does not reflect the actual business flow.

Instead, revise the responded solicitud detail view so that, after the reviewed response is displayed, the customer sees a clear next-step decision such as:

### ¿Cómo deseas proceder?

- Comprar con CRBOX  
  Short explanation: CRBOX purchases the product for the customer and applies the 5% service fee (minimum $5).

- Lo compraré por mi cuenta  
  Short explanation: the customer will purchase it directly, but the portal will track the expected package arrival to the Miami locker.

- optional secondary action: No continuar

--------------------------------------------------
State model refinement
--------------------------------------------------

Do not transition directly to completada.

Instead introduce separate states reflecting the two paths, for example:

- pendiente_compra_crbox
- pendiente_compra_cliente

or equivalent names that capture the two confirmed intents truthfully.

Only later should those progress toward states such as:
- paquete_esperado
- vinculada_a_paquete
- completada

completada should remain reserved for a later, more operationally final stage.

--------------------------------------------------
Portal response display
--------------------------------------------------

The reviewed response from response_json should be rendered in the portal detail view as proposed:
- confirmed shipping price
- availability
- delivery timeline
- conditions
- CRBOX message

That part is correct and should remain.

However, once this exists, the “Revisa tu correo” language should become secondary, not primary.
A better UX is:
- show the reviewed response content as the main panel
- include a smaller note such as:
  “También enviamos esta respuesta a tu correo.”

--------------------------------------------------
Package-linking model
--------------------------------------------------

Very important: linking an approved request / confirmed intent to a real package in Mis Paquetes must not override, mutate, or interfere with the existing CRBOX backend package record.

The legacy CRBOX package record remains the operational source of truth.

The quote/request system should instead create an overlay relationship or metadata layer:
- the approved request creates an expected-package record (or equivalent frontend/backend overlay object)
- when a real package later appears in Mis Paquetes, the system attempts to link it
- if linked, the UI shows a relationship such as:
  - “Relacionado con tu solicitud SCB-XXXX”
  - product name the user expected
- but without changing the original CRBOX backend package data

--------------------------------------------------
Linking recommendation
--------------------------------------------------

Please design the linking model as hybrid, not purely manual and not purely automatic.

### Strong automatic match
Auto-link only on strong signals, especially:
- exact tracking number match

### Assisted match
When the match is plausible but not certain:
- suggest the possible match in UI or admin
- require confirmation before finalizing the link

### Internal/manual fallback
If no strong match exists:
- allow an internal/manual admin-side link as a fallback

Do not auto-link only on weak heuristics like similar weight or similar declared value alone.

--------------------------------------------------
Expected package UX
--------------------------------------------------

When the customer chooses Lo compraré por mi cuenta, create a visible concept such as:

### Paquete esperado
This should communicate:
- this is not yet a received package
- this is an expected package tied to an approved request
- once CRBOX registers the package in Miami, the system will attempt to relate it automatically

Friendly copy example:
> “Cuando CRBOX registre el paquete en Miami, intentaremos relacionarlo automáticamente con esta solicitud.”

Also consider allowing the customer to optionally add a tracking number later, because that would make automatic linking much more reliable.

--------------------------------------------------
What should still proceed from the current plan
--------------------------------------------------

These parts of the current proposal are still correct and should proceed:

1. Render response_json natively in solicitud.html
2. Populate active badge counts in mis-solicitudes.html and dashboard.html
3. Use a dedicated portal-auth endpoint rather than broadening the admin-only status endpoint

But the endpoint and state transition should be revised to reflect the two-path intent model above rather than a single respondida -> completada confirm action.

--------------------------------------------------
Please revise the task accordingly
--------------------------------------------------

Please update the portal-side next-step flow with:
- two explicit customer intent paths
- a refined state model
- expected-package concept
- non-destructive overlay linking approach
- hybrid package linking strategy

This will make the portal experience much more faithful to the actual CRBOX business model and much more intuitive for the customer.
