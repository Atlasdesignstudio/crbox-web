# Task — Real per-package invoice upload flow in Mis Paquetes + visibility in Mis Facturas

## What & Why
The new client portal already has real package data in Mis Paquetes, and we have now verified the exact legacy backend flow for invoice upload. The missing piece is to make invoice upload fully usable and easy across the new portal experience:

- from Mis Paquetes
- in both table view and card view
- on desktop and mobile
- with a polished, premium-feeling form
- and with invoice visibility reflected clearly in Mis Facturas

This is not just a backend wiring task. It is also a UX completion task.

The goal is:

1. users can easily identify which packages still need an invoice
2. users can upload an invoice from any package row/card
3. the upload uses the real verified legacy flow
4. uploaded invoices are reflected back in package state
5. users can later see those invoices inside Mis Facturas, including the package they belong to and the package state where truthfully available

This must be implemented as a real production-quality flow, not a mock, not a redesign, and not a parallel backend.

---

## Verified production backend contracts

### Step 1 — upload invoice file
POST https://crbox.cr/wp-json/crbox/v1/saveBill

Content-Type: multipart/form-data

Form-data fields
- email
- invoice (binary file)
- wr_id

Verified response shape
- file
- url
- type

Example response includes a real uploaded file URL like:
- https://crbox.cr/wp-content/uploads/2026/04/FacturaCompra-WR606011.pdf

### Step 2 — create purchase bill record
POST https://clients.crbox.cr/api/crboxwebapi/postcreatepurchasebill

Content-Type: application/x-www-form-urlencoded

Payload fields
- ClientInvoiceText
- Descripcion
- FileLocation
- Monto
- NumeroFactura
- WRId

Verified success response
- StatusResult = "OK"
- Message = "La factura fue almacenada satisfactoriamente"

### Step 3 — refresh package list
After success, the live site refreshes package data through:
GET /api/crboxwebapi/getuserpackages/...

The refreshed package response reflects invoice state through real fields such as:
- idwarehousereceipt
- montofactura
- descripcionfactura
- invoicesCount
- impresoFactura

This means the new portal must also update from real backend refresh, not from optimistic local-only state.

## Critical identifier
The correct package-level identifier is:

- idwarehousereceipt

This exact value must be used as:
- wr_id in saveBill
- WRId in postcreatepurchasebill

Verified example:
- package row has idwarehousereceipt = 606011
- upload call uses wr_id = 606011
- bill-create call uses WRId = 606011

Do not derive another identifier. Do not use tracking number as the backend linkage key.

## Deterministic technical rules
These are not optional; use them to avoid guessing.

1. Invoice presence source of truth
   - Use invoicesCount > 0 as the primary source of truth for whether a package already has an invoice.
   - montofactura != null may be used only as a defensive fallback if needed, but not as the primary rule.

2. ClientInvoiceText format
   - Build ClientInvoiceText using the same simple legacy shape:
   - <p>${description}</p>
   - Do not introduce a rich-text editor requirement if not necessary.

3. Package refresh after success
   - After successful saveBill + postcreatepurchasebill, refetch package data using the existing package loader for the current filters/state/date window.
   - Do not rely only on optimistic local mutation.
   - Do not hard-reload the whole page unless there is no cleaner option.

4. Mis Facturas linkage truthfulness
   - Do not invent package linkage in Mis Facturas.
   - Only show linked package/receipt/status there if that information is truly available from current real backend responses.
   - If the current invoices backend does not expose enough linkage information, show only truthful invoice data and do not fabricate package context.

5. File validation
   - Allow the same types the legacy UI allows: PDF and common image formats.
   - Validate before submit.
   - Show a clear user-facing error if the file type is invalid.
   - If there is an existing known max-size rule in current code/contracts, reuse it. Do not invent arbitrary limits unless already enforced somewhere real.

- modal/form must fit the viewport

The form should:
- validate required fields before submit
- show simple helper text for file types
- show loading state while uploading
- show clear success/error feedback
- not use a heavy or old-looking UI
- preserve required field names/meaning

## Success and failure handling

### Success path
If both backend calls succeed:
- refetch packages
- rerender package state from real backend data
- update invoice action accordingly
- make the invoice discoverable in Mis Facturas through the real backend data flow

### Failure path A — file upload fails
If saveBill fails:
- do not call postcreatepurchasebill
- show a clear error
- do not mark the package as invoiced
- keep the user in a recoverable state

### Failure path B — bill create fails after upload
If saveBill succeeds but postcreatepurchasebill fails:
- do not mark the package as invoiced
- show a clear error
- do not fake success
- keep enough debug visibility in logs for diagnosis

### Failure path C — backend write succeeds but refetch fails
If upload/create succeeds but package refresh fails:
- show a partial-success style message
- do not invent final invoice state
- encourage refresh/retry
- avoid showing wrong post-upload status until real backend data is loaded
