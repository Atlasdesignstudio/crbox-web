Absolute safety guardrails

1. Use only the dev/test database

Only connect to:

MYSQL_DATABASE=crbox_dev1

Do not connect to, inspect, query, switch to, or use:
- `CrBox`
- `crboxfabric`
- any production database
- any other schema found on the server

If the connection reports any active database other than `crbox_dev1`, stop immediately and report the mismatch.

2. Do not touch existing systems

Do not change, restart, reconfigure, test, patch, or interact with:
- the old website
- the current internal/backoffice tool
- the legacy API
- AWS RDS settings
- AWS security groups
- firewall rules
- database users
- database passwords
- grants/permissions
- certificates
- production endpoints

This task is only a Replit backend read-only discovery against `crbox_dev1`.

3. Read-only means read-only

Even if the database user technically has write permissions, treat this task as read-only.

Forbidden SQL/actions:
- INSERT
- UPDATE
- DELETE
- DROP
- ALTER
- CREATE
- TRUNCATE
- GRANT
- REVOKE
- LOCK TABLES
- migrations
- seed scripts
- stored procedure execution with side effects
- account creation
- profile edits
- invoice uploads
- package edits
- package status edits
- permission changes
- user creation

Allowed queries only:
- SELECT DATABASE();
- SELECT VERSION();
- SHOW TABLES;
- SHOW COLUMNS FROM `<table>`;
- DESCRIBE `<table>`;
- SELECT COUNT(*) FROM `<table>`;

If you need any query outside this list, stop and ask first.

4. Avoid performance impact

Do not run heavy scans.

Do not run:
- SELECT * FROM large tables
- unbounded SELECT queries
- full table exports
- joins across large tables
- ORDER BY on large tables
- expensive aggregations
- schema modifications
- queries that can lock or stress the database

For row counts, use the existing count endpoint only. If a table appears very large, mark it as large and avoid further sampling.

5. Do not expose sensitive data

Do not return raw customer records, names, emails, phone numbers, addresses, IDs, invoices, tracking numbers, payment data, or personal data.

Use only:
- table names
- column names
- row counts
- schema structure
- inferred relationships
- masked examples only if absolutely necessary

If limited samples are needed later, ask first and mask values.

6. Be careful with conclusions

In the report, clearly separate:
- confirmed facts from schema
- strong hypotheses
- weak hypotheses
- unknowns / needs dev validation

Do not claim a table powers a module unless the evidence supports it. Use confidence levels.

7. Legacy API comparison

Do not call or stress the live legacy API during this task unless explicitly approved.

For now, compare against:
- known legacy endpoint contracts
- existing code references
- current frontend/backend API usage

Do not perform live stress testing or repeated calls against the legacy API.

8. Secrets safety

- MYSQL_PASSWORD

- credentials
- secret values

9. Final safety confirmations
