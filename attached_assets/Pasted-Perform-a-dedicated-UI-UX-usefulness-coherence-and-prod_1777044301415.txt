Perform a dedicated UI/UX usefulness, coherence, and product-quality audit of the CRBOX client portal implementation that now sits on top of the real backend.

This is a standalone refinement pass focused on the user experience quality of the portal, not just on backend truthfulness.

Do not treat this as a redesign.
Do not reinvent the design system.
Do not change the overall visual direction, structure, or style of the portal.
Instead, treat this as a high-level product polish + usefulness audit that improves the existing portal using the real data already available.

# Goal
Review the entire client portal as a user-facing product and refine it so that:

- the most useful real data is surfaced clearly
- less useful or overly technical data is not over-exposed
- information hierarchy feels intentional
- all pages feel smooth, clean, premium, and coherent
- the portal feels like one unified product, not a set of disconnected screens
- the existing UI is used in the best possible way without redesigning it

# Important framing
This is not a truthfulness-only pass.
This is not a data-mapping-only pass.
This is a user-value and UX coherence pass.

Assume the portal is already being wired to real backend data. Your job is to review whether the portal is making the best product use of that real data.

# Pages to review as one coherent portal
Audit all current client portal pages together, including:
- dashboard
- mis-paquetes
- mis-facturas
- mi-cuenta
- shared header / portal shell / navigation states

Review them not as isolated files, but as one connected experience.

# What to evaluate

## 1. Are we surfacing the most useful real data?
For every page and module, evaluate whether the current UI is surfacing the most meaningful backend fields for the user.

Use this rule:
If a real backend field is:
- semantically clear,
- genuinely useful to the client,
- and can fit into the current UI without redesign,

then it should be surfaced in the most appropriate place.

Examples of appropriate places:
- summary card
- table/list column
- card metadata line
- detail modal
- secondary detail section
- not shown at all, if it adds noise

Do not assume that every confirmed backend field must be shown.
The goal is usefulness, not maximum density.

## 2. Are we over-surfacing or under-surfacing information?
Audit both risks:

### Under-surfacing
The UI may currently hide real useful data that should help the user, especially in:
- Mis Paquetes
- Mis Facturas
- dashboard summaries
- detail modals

### Over-surfacing
The UI may also show too much low-value or confusing information in overview areas.

Decide what belongs in:
- overview
- table/list
- compact metadata
- modal/detail
- nowhere

## 3. Is the information hierarchy good?
Review whether each page presents information in the right order for the user.

Examples:
- what should the user notice first?
- what is the key status/action?
- what is supporting metadata?
- what belongs in a modal instead of the main surface?

You are allowed to improve information prioritization and placement within the existing structure, but not redesign the page.

## 4. Do all portal pages feel visually and behaviorally consistent?
Audit the portal as one product experience.

Check for consistency in:
- spacing rhythm
- density of cards/tables
- badge usage
- loading states
- empty states
- error states
- title/subtitle structure
- CTA placement
- detail modal tone
- use of metadata lines
- how much information is shown per surface

The output should feel smooth, intentional, elegant, and cohesive.

## 5. Does the portal feel polished?
Review whether the current implementation has rough edges such as:
- awkward loading transitions
- inconsistent empty states
- too much visual clutter
- too little useful context
- mismatched card density between pages
- labels that are technically correct but not user-friendly
- blocks that feel unfinished even if technically wired
- sections that are truthful but not very helpful

Improve the portal so it feels refined and client-ready.

## 6. Improve clarity, but do not invent meaning
If a real backend field is useful but the current label or placement makes it unclear, improve the label or placement.

However:
- do not invent semantics
- do not rename fields in a misleading way
- do not imply business meaning the backend does not actually support

## 7. Respect the difference between overview and detail
A major part of this audit is deciding where data belongs.

Use these principles:
- overview surfaces should prioritize fast scanning and user action
- detail surfaces should carry denser operational metadata
- tables should favor compare-able, decision-useful fields
- modals should absorb complexity that does not belong in overview

Do not dump every real field into the main page just because it exists.

## 8. Shared portal polish
Review the shared experience across all pages:
- auth-aware header states
- transitions from loading → loaded
- how user identity appears
- navigation clarity
- consistency of “real / neutral / próximamente” states
- perceived smoothness of the portal as a whole

# Constraints
- No redesign
- No new design system
- No major layout reinvention
- No changing public pages
- No inventing unsupported backend semantics
- No reintroducing fake data just because it looks better
- No adding noisy fields just because they exist

# What you should do
1. Audit the current portal page by page and also holistically.
2. Identify where useful real data is being underused.
3. Identify where the UI is too dense or too sparse.
4. Refine the information hierarchy using the existing UI structure.
5. Improve consistency across dashboard / mis-paquetes / mis-facturas / mi-cuenta.
6. Keep the portal smooth, clean, aesthetic, and coherent.
7. Make targeted product-quality improvements, not redesign moves.

# Deliverable
At the end, report back with:

1. Which useful real data was not being leveraged before and is now surfaced
2. Which real fields were intentionally kept hidden and why
3. Which UX/UI coherence improvements were made across the portal
4. Which pages or modules were refined for smoother, cleaner, more helpful user experience
5. Any remaining UX limitations caused by missing backend support rather than UI issues

# Output standard
Be product-minded.
Be user-value-driven.
Be elegant and restrained.
Use the current design system well.
Make the portal feel complete, coherent, and genuinely useful.
