Act as a Senior Digital Analytics Architect + GA4/GTM Implementation Lead + CRO Measurement Strategist for CRBOX.

Your job is to audit, validate, complete, and improve the full Google Analytics / Google Tag Manager measurement setup for the CRBOX website and client portal in the most efficient, business-relevant and production-ready way possible.

This is not a passive audit. Treat this as an actionable analytics implementation sprint. Scan the entire project, understand the existing tracking setup, preserve what already works, identify what is missing, and implement the missing measurement layer correctly.

Context:
CRBOX has a public website and client portal experience, including pages such as:
- Homepage
- Servicios
- Cómo funciona
- Tarifas
- Calculadora
- Contacto
- Afíliate / registration
- Login
- Dashboard
- Mi cuenta
- Mis paquetes
- Mis facturas
- Cotizaciones / quote flow
- Any other public, portal or support page found in the project

The goal:
Make sure GA4 / GTM is set up correctly and tracks the full customer journey in a clean, scalable and business-useful way.

Important:
Do not track random or meaningless interactions.
Do not create noisy events just because something is clickable.
Do not duplicate events.
Do not break existing tags.
Do not send personal data to analytics.
Do not track passwords, IDs, emails, phone numbers, addresses, tracking numbers, invoice numbers, customer names, casillero numbers or any other personal/customer-identifiable information.
Do not fake analytics.
Do not invent conversions.
Do not change visible UI, design, copy or business logic unless absolutely required for analytics instrumentation.

Primary objective:
Create a measurement setup that helps CRBOX understand:
- Which pages drive users toward affiliation/registration.
- Which pages drive users toward the calculator.
- Which pages drive users toward contact/WhatsApp.
- Which pages drive users toward quote requests.
- Where users drop off in registration.
- Where users drop off in login.
- Where users drop off in calculator usage.
- Where users drop off in quote request flow.
- How client portal users interact with packages, invoices, account and quote actions.
- Which CTAs and flows are producing meaningful business intent.
- Which pages are underperforming as entry points into key journeys.
- Which actions should be treated as conversions or micro-conversions.
- Which data is needed later for Tableau dashboards.

Core principle:
Track fewer things, but track the right things extremely well.

The analytics setup should be:
- Clean
- Consistent
- Useful
- Privacy-safe
- Funnel-ready
- Dashboard-ready
- CRO-ready
- Scalable
- Easy to debug
- Easy to connect to Tableau later

PHASE 2 — Measurement strategy

Create a clean event taxonomy before implementing.

Classify events into:

1. Core conversion events
These are the most important business outcomes:
- sign_up_submit_success
- login_success
- quote_request_submit_success
- calculator_result_view
- contact_form_submit_success
- whatsapp_click
- phone_click
- email_click

Only mark as success if there is real confirmation. Do not fire success events on button click alone.

2. Micro-conversion events
Important intent signals:
- calculator_start
- calculator_input_complete
- quote_request_start
- quote_request_step_complete
- registration_start
- registration_step_complete
- contact_form_start
- package_search
- invoice_filter_use
- profile_edit_start
- profile_update_success
- password_recovery_submit

3. CTA navigation events
Track only meaningful CTAs:
- cta_click

Use parameters to describe:
- cta_text
- cta_location
- cta_destination
- page_type
- journey_stage
- user_state

Do not create separate event names for every CTA unless there is a strong reason.

4. Portal engagement events
Track only useful portal actions:
- portal_view
- package_filter_use
- package_refresh
- invoice_filter_use
- account_section_view
- quote_status_view
- logout

Do not track every tab hover, animation, menu open or meaningless UI interaction.

5. Error/friction events
Track meaningful friction:
- form_error
- login_error
- registration_error
- quote_request_error
- calculator_error
- api_error
- session_expired

Do not send raw backend error messages if they may contain sensitive information. Use sanitized error categories.

PHASE 3 — Required event naming standards

Use GA4-friendly lowercase snake_case event names.

Rules:
- Event names must be consistent.
- Use one reusable event with parameters when possible instead of many one-off events.
- Do not exceed GA4 limits.
- Do not send PII.
- Do not send full URLs containing sensitive query parameters.
- Do not send free-text user input.
- Do not send product descriptions typed by users unless explicitly anonymized and safe.
- Do not send email, phone, ID, address, invoice, package tracking, casillero number or user name.
- Use stable parameter names.

Recommended global parameters for most events:
- page_type
- page_path_clean
- page_section
- journey_stage
- user_state
- device_context
- language
- cta_text
- cta_location
- cta_destination
- form_id
- form_type
- flow_name
- step_name
- step_number
- status
- error_type
- shipping_mode
- quote_type
- account_type
- result_available
- portal_area

Important:
Any parameter that could contain PII must be removed, anonymized or replaced with a safe category.

PHASE 7 — Existing tags and GTM compatibility

The implementation must respect existing tags.

Do not blindly replace everything.

Steps:
1. Detect current GTM container snippet.
2. Confirm it appears once per page.
3. Confirm it is placed correctly.
4. Confirm noscript fallback exists where appropriate.
5. Detect existing dataLayer initialization.
6. Detect existing event pushes.
7. Consolidate inconsistent event names.
8. Prevent duplicate firing.
9. Preserve existing working events unless they are broken, duplicated, unsafe or useless.
10. If events need migration, add compatibility notes.

If GTM is the main tracking method:
- Implement clean dataLayer.push events.
- Do not hardcode GA measurement IDs in multiple places unless that is the current architecture.
- Keep the event layer clean so GTM can map events to GA4 tags.

If GA4 gtag is used directly:
- Avoid duplicate gtag and GTM firing.
- Prefer one source of truth.
- Document the current architecture and the recommended future architecture.
