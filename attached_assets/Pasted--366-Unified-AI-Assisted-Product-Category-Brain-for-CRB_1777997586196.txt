#366 - Unified AI-Assisted Product Category Brain for CRBOX Quote Funnel

## What & Why

We want to improve the product category selection experience across the entire CRBOX quote and calculator funnel.

Right now, the product category dropdown is a poor UX pattern. Users have to manually open a long dropdown, scroll through many options, guess which category applies to their product, and they do not clearly see what percentage of taxes that category usually pays.

This creates several problems:

- The user experience feels outdated.
- Users may choose the wrong category.
- The dropdown is hard to use on mobile.
- The category logic may be inconsistent across pages.
- The user does not understand why a product pays a certain percentage.
- The calculator logic may be limited by the current category list.
- Product URL analysis and manual category selection may not use the same logic.
- The public pages, client portal, internal sales tool, portal response, and email response may not explain category/tax logic consistently.

The goal is to replace or enhance the current category dropdown behavior with a unified AI-assisted product classification flow powered by a local CRBOX product brain.

The user should be able to type natural product descriptions such as:

- zapatos rosados
- Nike pink running shoes
- iPhone 15 Pro Max
- perfume Dior
- proteína whey
- power bank
- laptop gamer
- repuesto de carro
- audífonos Bluetooth
- maquillaje Sephora
- cargador USB-C
- air fryer
- smartwatch
- suplemento de magnesio

The system should understand what the product is, assign it to the best CRBOX category, show a simple user-friendly tax percentage explanation, and then pass that category/rate into the existing calculator logic.

Important: AI/Gemini should help interpret the product, but the full calculator math must stay in code. AI should not calculate the full quote. The calculator code should continue handling value, weight, volume, freight, fuel surcharge, handling, insurance, delivery, and final totals.

---

## Core Goal

Turn the category dropdown into a smart product understanding experience.

Instead of forcing the user to manually choose from a long dropdown, the system should work like this:

1. User types a product description or pastes a product URL.
2. User clicks “Analizar producto” or the existing “Analizar” action.
3. Gemini/local product intelligence identifies what the product likely is.
4. The system maps that product to the closest CRBOX local product category.
5. The UI shows:
   - Product detected
   - Category detected
   - Estimated tax percentage
   - Simple explanation
   - Any special requirement or review warning
6. The user can confirm it or manually change it.
7. The coded calculator uses that selected category and rate.

Example:

User enters:
“zapatos rosados”

The UI should show something like:

“Detectamos que este producto pertenece a Calzado. Esta categoría suele pagar aproximadamente 29% en impuestos. El monto final puede variar según el material, la suela y la revisión aduanera.”

Not:

“confidence: Media / DAI: 15 / law6946Rate: 1 / source: local_estimated”

The technical metadata can exist internally, but customer-facing copy must be human and simple.

---

## Main Rule

There must be ONE shared product/category/tax logic across all quote-related touchpoints.

Do not create separate category logic for different pages.

The same product should be classified the same way everywhere.

If the user enters “zapatos rosados” in the public calculator, public Cotizar page, client portal, or internal sales tool, it should resolve to the same local product category:

- Category: Calzado
- Orientative tax: approximately 29%
- User-friendly message about material/suela variation
- Same manual review behavior if applicable

## Architecture Direction

Use a hybrid architecture.

### 1. Local Product Brain

Create or improve a local CRBOX product brain.

Suggested file:

js/product-categories.js

This should be the local source of truth for:

- Product categories
- Product examples
- Keywords / aliases
- Estimated tax percentage
- Manual review flags
- Special risk flags
- User-friendly messages
- Admin notes
- Source label: local_estimated

The product brain does not need to list every exact sentence a user could write. Gemini is intelligent enough to interpret natural language. But the product brain must be broad enough and clear enough to provide the official CRBOX category/tax map.

### 2. Gemini / AI Interpretation Layer

Gemini may be used to interpret natural product descriptions, product URL titles, or ambiguous inputs.

Gemini’s role:

- Understand what the user is describing.
- Identify the likely product type.
- Detect if the product may have special requirements, such as battery, supplement, aerosol, chemical, drone, medical product, etc.
- Map the product to one of the local PRODUCT_CATEGORIES.

Gemini must not:

- Invent tax percentages.
- Invent customs rules.
- Invent official classifications.
- Calculate the full quote.
- Give final legal/customs decisions.
- Say something is illegal unless CRBOX has an explicit rule.

Gemini must select from the local CRBOX product brain.

### 3. Calculator Code

The calculator code remains responsible for math.

Calculator code handles:

- Declared value
- CIF logic if currently used
- Product value
- Weight
- Dimensions
- Volumetric weight
- Billable weight
- Freight
- Fuel surcharge
- Handling
- Insurance
- Delivery
- Taxes
- Multi-product calculation
- Consolidated calculation
- Final total

AI only provides:

- Product detected
- Category selected
- Tax rate/range
- Manual review flag
- Special requirement/risk flags
- Customer-friendly message
- Admin notes

## ProductCategoryUtils API

Expose a shared utility object:

window.ProductCategoryUtils = {
  search(query),
  analyzeProductInput(input),
  classifyWithGeminiIfNeeded(input, context),
  findBestMatch(query),
  getById(id),
  getByCode(code),
  getEstimatedRate(categoryIdOrQuery),
  isManualReviewRequired(categoryIdOrQuery),
  estimateCourierTax(productSearch, declaredValue, shippingCost, insuranceCost),
  toCSV()
}

All functions must be safe:

- Empty input should not crash.
- Unknown input should return manual review.
- Regulated products should return manual review.
- Gemini should never invent tax rates.
- Manual-review products should not produce final-looking automatic totals.

## Gemini Instruction

When Gemini is used, instruct it clearly:

“Interpret the product description or product page title and map it to the closest category from the provided PRODUCT_CATEGORIES list. Do not invent percentages, customs rates, official classifications, or final legal conclusions. Return the selected category id, detected product name, risk flags if any, and a short user-friendly explanation. If uncertain, return manual_review_required.”

Gemini can infer:

“Nike pink running shoes” = footwear / running shoes / Calzado

But the final category and percentage must come from PRODUCT_CATEGORIES:

Calzado → approximately 29%

## Product URL “Analizar” Action

The existing product URL analysis must use the same classifier.

When the user pastes a link and clicks “Analizar”:

1. Existing AI/link analysis extracts product title/details.
2. The detected title/details are passed into the shared product classifier.
3. The classifier maps the product to PRODUCT_CATEGORIES.
4. The UI shows the detected category and estimated tax percentage.
5. The coded calculator uses that category/rate.

Do not keep product URL analysis separate from manual product input.

Manual input and URL analysis must converge into the same product classification logic.

## User-Facing Messaging Rules

Customer-facing messages must be simple, human, and useful.

Avoid technical output such as:

- confidence: Alta
- DAI: 15
- ley6946Rate: 1
- source: local_estimated
- manual_review_required
- riskFlags: lithium_battery

Instead, show natural explanations.

### Example: Calzado

“Detectamos que este producto pertenece a Calzado. Esta categoría suele pagar aproximadamente 29% en impuestos. El monto final puede variar según el material, la suela y la revisión aduanera.”

### Example: Celulares

“Detectamos que este producto pertenece a Celulares. Esta categoría suele pagar aproximadamente 14% en impuestos. La estimación puede variar según el valor declarado y la validación final.”

### Example: Suplementos

“Este producto requiere revisión antes de cotizarlo automáticamente. Los suplementos pueden necesitar permisos o validación sanitaria, por eso nuestro equipo debe revisarlo antes de confirmar el costo final.”

### Example: Batería / Power bank

“Detectamos que este producto podría incluir una batería de litio. Estos productos suelen requerir revisión porque pueden tener condiciones especiales de transporte. En algunos casos pueden necesitar manejo especial o generar costos adicionales.”

### Example: Aerosol / químico

“Detectamos que este producto podría ser aerosol, inflamable o contener químicos. Este tipo de producto puede tener restricciones de transporte o requerir manejo especial. CRBOX debe revisarlo antes de confirmar si puede enviarse.”

### Example: Drone

“Detectamos que este producto parece ser un drone o equipo de control remoto. Estos productos pueden requerir revisión por batería, radiofrecuencia o regulación especial antes de confirmar la importación.”

### Example: Armas / tácticos

“Detectamos que este producto puede pertenecer a una categoría restringida, como armas, partes de armas, municiones o artículos tácticos. CRBOX debe revisarlo antes de confirmar si puede transportarse o importarse por courier.”

## Special Requirement / Risk Detection

Gemini should also help detect special requirements or risks from the product name, product URL, product title, or description.

Gemini should flag possible issues such as:

- Lithium battery / rechargeable battery
- Power bank / power station / jump starter
- Aerosol
- Flammable product
- Chemical product
- Paint, glue, resin, solvent, adhesive
- Supplement, vitamin, protein, pre-workout, herbal product
- Medicine or medical product
- Food or beverage
- Alcohol
- Tobacco, vape, nicotine
- Plants, seeds, soil, fertilizer, pesticide
- Drone or remote-controlled aircraft
- Radio, telecom, satellite, antenna, communication device
- Medical device or medical supplies
- Weapon, firearm part, ammunition, tactical knife, restricted item
- Product that may be illegal, restricted, or not suitable for courier
- Commercial quantity or repeated units that may change treatment

When a risk is detected, the user-facing answer must not be vague.

Do not only say:

“Puede requerir revisión.”

Instead, explain:

1. What was detected.
2. Why it matters.
3. What CRBOX needs to do next.
4. Whether it may affect shipping, timing, cost, or eligibility.

Use careful but specific language:

- “suele requerir revisión”
- “puede requerir permiso”
- “puede tener restricciones de transporte”
- “puede generar costos adicionales”
- “CRBOX debe validarlo antes de confirmar”

Avoid definitive legal claims unless CRBOX has an explicit rule:

- “es ilegal”
- “está prohibido”
- “garantizado que requiere permiso”
- “definitivamente no se puede importar”

If risk is detected, return internal metadata like:

{
  riskFlags: ["lithium_battery", "special_transport"],
  manualReviewRecommended: true,
  automaticEstimateAllowed: false,
  customerMessage: "Detectamos que este producto podría incluir una batería de litio. Estos productos suelen requerir revisión porque pueden tener condiciones especiales de transporte. En algunos casos pueden necesitar manejo especial o generar costos adicionales."
}

The customer should see the human message.

The admin/sales team can see the technical flags.

## Quote and Calculator Math

The AI/category system only provides:

- category id
- display category
- estimated tax percentage or range
- manual review flag
- risk flags
- customer message
- admin notes

The existing calculator code remains responsible for:

- declared value
- CIF
- weight
- volumetric weight
- billable weight
- freight
- fuel surcharge
- handling
- insurance
- delivery
- total quote
- multi-product/consolidated calculation

Do not move calculator math into Gemini.

---

## Quote Payload Requirements

Each product in a quote should store structured category information:

{
  productName,
  originalProductInput,
  analyzedProduct,
  selectedCategoryId,
  selectedCategoryName,
  estimatedTaxRate,
  estimatedTaxRange,
  daiRate,
  vatRate,
  law6946Rate,
  needsReview,
  regulatedProduct,
  automaticEstimateAllowed,
  riskFlags,
  customerMessage,
  source: "local_estimated",
  units,
  declaredValue,
  weight,
  dimensions
}

---

## Customer-Facing Result Requirements

Wherever the category appears to the customer, show:

- Product detected
- Category detected
- Friendly estimated tax percentage/range
- Friendly special warning if relevant
- Simple explanation that final amount may vary

This must appear consistently in:

- Public Cotizar page
- Public Calculadora page
- Client portal Cotizaciones
- Portal quote response
- Customer quote response email

---

## Admin-Facing Result Requirements

In the internal CRBOX sales tool, show:

- Original input
- Detected product
- Suggested category
- Category used for quote
- Estimated rate/range
- DAI / IVA / Ley 6946
- Risk flags
- Review reason
- Admin notes
- Customer-friendly message
- Ability to override category

If sales overrides the category, clearly show:

- Categoría detectada por el sistema
- Categoría usada para cotización

---

## Do Not Do

Do not:

- Break existing calculator math.
- Break existing quote submissions.
- Break existing quote responses.
- Break existing admin response_json behavior.
- Create separate category lists per page.
- Force users to scroll through a long dropdown.
- Let Gemini invent rates.
- Let Gemini calculate the full quote.
- Present estimates as official final customs liquidation.
- Automatically calculate regulated/manual-review products as final.
- Hide manual review requirements.
- Show technical metadata to customers.
- Remove admin override capability.
- Depend on external APIs.

---

## Acceptance Criteria

- Category dropdown UX is replaced or enhanced with natural product search/classification.
- Users can type natural language product names.
- Product URL “Analizar” uses the same classifier.
- Gemini can interpret product names but must map to PRODUCT_CATEGORIES.
- Tax percentages come only from PRODUCT_CATEGORIES.
- Calculator math stays in code.
- Public Cotizar uses this logic.
- Public Calculadora uses this logic.
- Client portal Cotizaciones uses this logic.
- Internal CRBOX sales calculator uses this logic.
- Portal quote response uses this logic.
- Customer quote email uses this logic.
- User-facing messages are friendly and not technical.
- Special requirement warnings are clear and useful.
- Admin can see technical details and override category.
- Manual-review products do not show final automatic totals.
- Existing CRBOX functionality remains intact.
- No duplicated category logic remains.
- No console errors.
