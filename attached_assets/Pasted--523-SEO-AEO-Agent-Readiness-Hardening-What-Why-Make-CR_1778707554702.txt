#523 - SEO/AEO/Agent Readiness Hardening

What & Why  
Make CRBOX machine-readable, LLM-discoverable, crawler-efficient, and agent-ready across its public layer.

The foundation is already in place: robots.txt, sitemap.xml, llms.txt, and noindex handling on private pages. This task builds the next missing layers:

- a public structured JSON context file
- safe public API endpoints
- JSON-LD schema improvements on every public page
- complete per-page metadata
- basic performance wins
- accessibility essentials
- public/private truthfulness checks
- a future-agent/MCP architecture document

The end goal is that Google can reliably crawl and index the correct public pages, LLMs can answer core CRBOX questions from structured public data without guessing, and future AI agents have a documented, safe interface architecture to build on.

This is an implementation task, not just an audit.

Done looks like

- /ai-context.json is served correctly, valid JSON, public-only, schema-versioned, and parseable by any agent.
- Public read-only API endpoints exist in server.py:
  - /api/public/overview
  - /api/public/services
  - /api/public/how-it-works
  - /api/public/faqs
  - /api/public/contact
  - /api/public/rates-guidance
- All public endpoints return structured JSON sourced from knowledge/crbox-kb.json and/or existing approved public knowledge. No private data, no fake data.
- Public endpoints are CORS-open, cacheable, and safe for agents to consume.
- Every public HTML page has:
  - unique <title>
  - unique <meta name="description">
  - correct <link rel="canonical">
  - matching Open Graph tags
  - matching Twitter/X tags
  - lang="es" on <html>
- Every public page has appropriate JSON-LD:
  - Organization and WebSite where appropriate
  - WebPage on every public page
  - FAQPage only where real visible FAQ content exists
  - Service schema only where real visible service content exists
  - no fake reviews, no fake ratings, no fake prices, no unsupported claims
- Key public images have width and height attributes.
- Below-the-fold images use loading="lazy" and decoding="async".
- Hero/LCP images are not lazy-loaded.
- Non-critical scripts use defer where safe.
- Icon-only buttons have aria-label.
- Forms have proper label/input relationships.
- Active nav links use aria-current="page".
- Every public page has a <main> landmark.
- No fake/demo/private content appears in public HTML, schema, public JSON, or public JS.
- docs/crbox-agent-architecture.md exists and defines the future CRBOX agent architecture clearly.
- llms.txt and ai-context.json reference all relevant public machine-readable resources.
- Existing UI, layout, backend contracts, authenticated flows, analytics, and portal behavior remain unchanged.

Out of scope

- Redesigning existing UI or page layouts
- Changing authenticated endpoints
- Changing login flow
- Changing registration flow
- Changing portal logic
- Implementing a live MCP server
- Changing GTM or analytics configuration
- Replacing or generating new image assets
- Making cotizar.html publicly indexable
- Any changes to private portal pages beyond confirming noindex/private treatment is correct
- Backend migrations or risky architecture changes

Core rules

1. Do not fake anything.  
Do not invent prices, services, delivery times, testimonials, ratings, customer names, package data, invoice data, branches, policies, or business facts.

2. Use knowledge/crbox-kb.json as the primary source of truth.  
Where possible, public JSON, API endpoints, schema, and llms.txt should all derive from the same approved knowledge source to avoid inconsistencies.

3. Do not expose private data.  
No customer names, emails, package IDs, invoice data, addresses, account details, tokens, backend secrets, or private operational data should appear in public files, schema, endpoints, or llms.txt.

4. Do not break existing behavior.  
Preserve the current visual design, authenticated flows, portal functionality, calculators, registration, login, forms, and navigation.

5. Act safely.  
Implement everything that is safe in the current codebase. Leave only blocked or risky items as follow-ups.

Steps

1. Create /ai-context.json

Create a complete schema-versioned JSON file served at /ai-context.json.

Source facts from:
- knowledge/crbox-kb.json
- existing llms.txt
- existing approved public website content

Include:

- schemaVersion
- lastUpdated
- brand
- publicResources
- restrictedResources
- services
- howItWorks
- calculatorOrEstimate
- contact
- faqs
- agentGuidance

Requirements:

- valid JSON
- public-only data
- no private data
- no fake data
- no unsupported pricing or delivery claims
- clear limitations for rates/estimates
- cache header: Cache-Control: public, max-age=3600

2. Add public API endpoints

Add six safe GET endpoints in server.py:

- /api/public/overview
- /api/public/services
- /api/public/how-it-works
- /api/public/faqs
- /api/public/contact
- /api/public/rates-guidance

Each endpoint should:

- return structured JSON
- source data from knowledge/crbox-kb.json
- expose only public information
- include no customer/private/admin data
- include no secrets or internal-only URLs
- include CORS header: Access-Control-Allow-Origin: *
- include cache header: Cache-Control: public, max-age=3600
- be rate-limit-exempt if the app has rate limiting
- fail gracefully with safe error messages
- avoid exposing stack traces

Add these endpoints to the public resources section of llms.txt and ai-context.json.

3. Audit and fix per-page metadata

For each public HTML page currently intended to be public/indexable, including:

- index.html
- servicios.html
- como-funciona.html
- tarifas.html
- calculadora.html
- contacto.html
- afiliate.html
- privacidad.html
- terminos.html

Ensure:

- unique descriptive <title>
- unique meta description
- correct canonical URL
- og:title
- og:description
- og:url
- og:image only if the referenced asset exists
- twitter:card
- twitter:title
- twitter:description
- twitter:image only if the referenced asset exists
- lang="es" on <html>

Guidelines:

- titles should be clear and useful, not keyword-stuffed
- meta descriptions should explain the page purpose clearly
- canonical URLs must match the production public URL
- do not change body content as part of this step unless necessary for schema/landmark/accessibility work

4. Improve JSON-LD structured data

Audit all existing JSON-LD on public pages.

Implement or improve:

- Organization schema from real CRBOX public data
- WebSite schema where appropriate
- WebPage schema on every public page
- BreadcrumbList where breadcrumbs exist or are appropriate
- FAQPage only where visible Q&A exists in the HTML
- Service schema on service-related pages using real visible service descriptions
- ContactPoint only if real public contact information exists

Strict rules:

- no fake Review schema
- no fake AggregateRating
- no fake Product schema
- no fake prices
- no unsupported delivery promises
- no schema that contradicts visible page content
- no private portal/client data
- no duplicated conflicting schema blocks

Validate JSON syntax after implementation.

5. Performance quick wins

On all public pages:

- add width and height to <img> tags that are missing them
- add loading="lazy" decoding="async" to below-the-fold images
- do not lazy-load first hero/LCP image
- use fetchpriority="high" for the main LCP image only if appropriate
- add defer to non-critical scripts where safe
- do not defer scripts that must run before dependent scripts
- avoid changing build tools or bundling
- avoid risky dependency rewrites

Document any larger performance recommendations separately, especially if they require:

- replacing CDN Tailwind
- replacing Font Awesome
- image compression/replacement
- bundling changes
- hosting/CDN configuration

6. Accessibility essentials

On public pages:

- add aria-label to icon-only buttons
- ensure hamburger/search/close buttons have accessible names
- add for/id pairs for form labels where missing
- add autocomplete attributes to common form fields where safe
- add aria-current="page" to the active nav link
- ensure each public page has a <main> landmark
- preserve visible design
- do not redesign components
