I want a targeted UX + reliability improvement pass on the product URL analysis flow in the public Cotizar page.

This is specifically about the field:
- URL del producto
and the Analizar action.

Right now there are two problems:

1. Poor loading feedback / weak user guidance
When the user pastes a product URL and clicks Analizar, there is no clear feedback that the system is actively processing the URL. The user does not know:
- whether it is loading
- whether it is working
- whether they should wait
- or whether the action failed

2. The crawl / extraction reliability is not good enough
The current analysis flow is not consistently succeeding when trying to read product pages. I want this reviewed seriously, not just cosmetically. If the current implementation depends too heavily on a weak LLM-only path or a brittle fetch/crawl path, I want it improved in the smartest way possible.

This does not mean redesigning the whole Cotizar page.  
It means improving the URL analysis experience and reliability.

---

## Main goals

### A. Improve the UX of the Analyze action
When the user clicks Analizar, the interface should clearly communicate that the system is working.

The user should never feel like:
- nothing happened
- the button is dead
- the page froze
- or they need to click again

### B. Improve the extraction reliability
Audit the current product URL analysis pipeline and improve it so it is more robust for real retailer/product pages.

I want you to inspect whether the current system is using:
- simple fetch only
- HTML parsing only
- LLM-only extraction
- a weak crawl path
- or some combination of those

Then improve it with the most pragmatic architecture possible.

If that means:
- better scraping/parsing
- fallback extraction layers
- retailer-aware parsing
- metadata extraction before LLM use
- structured extraction first, LLM second
- or even a more agentic pipeline

then do that only if it makes the system more reliable without overengineering it.

## UX requirements for the loading / analysis state

When the user clicks Analizar:
- the button should enter a clear loading state
- the user should see that analysis is in progress
- the field/module should communicate that this may take a moment
- multiple accidental clicks should be prevented

### Good examples of what I want
- button changes to a loading state such as:
  - “Analizando…”
  - or “Obteniendo datos…”
- spinner / loader appears
- helper text appears under the URL field, such as:
  - “Estamos analizando el enlace y completando los datos del producto.”
- inputs or button state clearly reflect that processing is underway
- once completed, the user gets a clear success or failure state

### Important
Do not make this noisy or heavy.
It should feel:
- elegant
- calm
- modern
- trustworthy
- native to the current UI

---

## UX requirements for success / failure feedback

### On success
Make it clear that the product data was found and fields were filled.

For example:
- subtle success message
- or a “Datos completados” helper state
- or another elegant confirmation

### On failure
Do not leave the user in ambiguity.

If the system cannot extract the product:
- show a clear, friendly error state
- explain that they can still complete the fields manually
- ideally indicate that some links may not allow automatic reading

Example direction:
- “No pudimos completar los datos automáticamente. Puedes ingresarlos manualmente.”

### Important
Failure should not feel like the form is broken.

## Preferred technical direction
I do not want a naive “just ask an LLM to guess from the URL” flow.

A better direction is something like:
1. fetch page / retrieve structured metadata first
2. parse strong signals first:
   - title
   - price
   - currency
   - images
   - weight / dimensions if present
   - JSON-LD / schema data
   - Open Graph / product metadata
3. use retailer-aware extraction where possible
4. only then use LLM as a fallback / cleanup / interpretation layer if needed

If an agentic or multi-step approach genuinely improves robustness, that is acceptable — but keep it pragmatic.

I do not want unnecessary complexity just for the sake of saying “agent”.

---

## Important architectural constraint
Do not break the current form.

The user must always be able to:
- paste a URL
- try analysis
- and still continue manually if the system cannot extract the product

The analysis should be an enhancement, not a dependency that blocks the form.

## Retailer reality
Assume the system should handle links from places like:
- Amazon
- Walmart
- eBay
- common online stores

Some sites block bots or have inconsistent HTML.  
The system should degrade gracefully:
- do its best to extract
- and if it cannot, fail clearly and let the user continue
