I want you to do a data-leverage audit of the current CRBOX client portal and backend-connected data we already have, and identify what valuable information is not being surfaced well enough today, both for the client-facing experience and for internal/business benefit.

This is not a request to invent new backend data or fake insights.
It is a request to think deeply about the real data we already have access to and determine:

1. what should be surfaced to the client because it would genuinely improve clarity, trust, usability, and perceived value
2. what can be used internally for product/business/operations benefits even if it is not necessarily shown directly in the UI
3. what relationships between existing data points are currently under-leveraged

Core goal

Audit the current real connected data in the portal and think:

* what are we already receiving that is useful?
* what are we not leveraging enough?
* what can be derived truthfully from real current data?
* what would be genuinely valuable for the user to understand better?
* what would help CRBOX internally if used more intelligently?

I want product thinking, not just a field inventory.

⸻

Important principles

* Do not invent new data.
* Do not infer business semantics too aggressively if they are not defensible from the payload.
* Do not propose fake dashboards or vanity metrics.
* Stay grounded in the real shapes we already know from the portal:
    * packages
    * bills / facturas
    * items/recibos inside facturas
    * profile / sucursal / account info
    * any other currently connected real endpoint data already in the project
* Separate clearly:
    * directly showable to the client
    * truthfully derivable
    * internally useful but not necessarily client-facing
    * ambiguous / should not be surfaced yet

⸻

Example of the type of thinking I want

Not this:

* “show more fields”

But this:

* if a factura contains multiple recibos/items, that may indicate a consolidated shipment or grouped billing event, so maybe the UX should surface that in a more user-friendly way
* if packages share a MAWB/HAWB or factura grouping, maybe the portal could explain that relationship better
* if there are repeated carriers, repeated stores, repeated branch selections, or repeated delivery patterns, maybe there is useful summarization or client education value there
* if the user has unresolved setup/profile data, maybe existing data can be used to guide activation more intelligently
* if there are fields useful for operations, support, retention, or upsell, identify them too

I want this level of product reasoning.

⸻

What I want you to analyze

Please audit the currently available real data and think in these categories:

1. Client-facing value

What current data could be surfaced better to the client in the portal because it would help them:

* understand their shipments better
* understand facturas better
* understand grouped vs individual orders better
* trust the system more
* reduce confusion
* reduce support needs
* feel the portal is more intelligent and useful

For each idea, explain:

* what real data it uses
* whether it is directly available or derived
* why it is useful to the client
* where it would best live in the UX (dashboard, package detail, factura detail, card, summary, timeline, badge, grouped section, etc.)

2. Internal / business / operational value

What current data could be leveraged internally for benefit, even if not shown directly to the client?

Examples of the kind of benefit I mean:

* support prioritization
* operational visibility
* client segmentation
* retention triggers
* lifecycle nudges
* identifying high-value users
* identifying friction patterns
* identifying incomplete setup states
* identifying consolidation/grouping patterns
* identifying repeat behaviors or customer patterns

For each idea, explain:

* what data it depends on
* whether it is directly available or derived
* what internal value it creates
* whether it should remain internal only or partially client-facing

3. Relationships between entities that are underused

Look for relationships between:

* packages and facturas
* facturas and recibos/items
* packages and MAWB/HAWB
* package status + billing
* sucursal + shipment flow
* carriers + stores + invoice groupings
* profile state + operational readiness

I strongly suspect there are useful relationships already present that are not being expressed well in the product today.

4. Truthfulness boundary

For every suggestion, be explicit about:

* what is safe to show
* what is safe to derive
* what is too ambiguous and should not be surfaced
* what would require future backend confirmation

I want you to be ambitious but disciplined.
