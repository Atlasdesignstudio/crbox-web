#316 - Full Quote Funnel Upgrade — Public Quote Page, Client Portal Quote Flow, Internal Sales Quoting Tool & Quote Response Experience

## Important Instruction

Do not implement this as isolated UI patches.

Treat this as a connected quote funnel audit and upgrade. First map the current quote flow from:

1. Public quote submission
2. Client portal quote submission
3. Internal sales review/calculation
4. Sales-generated customer response
5. Customer-facing email response
6. Client portal quote response display, when the quote came from the portal

Then make changes in a way that keeps calculation logic, category data, unit conversion, quote payloads, sales response data, and customer-facing output consistent across the whole system.

The quote response is part of the product experience, not only an email. For client-portal quotes, the response must also be rendered inside the portal with the same full disclosure and calculation detail, including the consolidated multi-product calculation UI when applicable.

---

## What & Why

Improve the full CRBOX quote funnel across three connected surfaces:

1. Public quote page / Cotizar page
2. Client portal quote flow
3. Internal sales quote review/calculation tool

The current public quote page is the most updated experience and should become the reference. The client portal quote flow currently feels more outdated and should be upgraded to match or improve upon the public quote experience, while taking advantage of the fact that logged-in clients already have known profile data such as locker number, email, name, phone, preferred branch/delivery method, and addresses.

The internal sales quoting tool must also be improved so sales agents can calculate, review, adjust, respond to, and explain quotes more accurately and efficiently.

This task is not only a UI update. It is a full quote workflow improvement focused on:

- Better product category selection.
- More complete tariff/tax category coverage.
- Multi-product quote support.
- Consistent calculator logic across public, portal, and internal tools.
- Better unit conversion experience.
- Clearer quote breakdowns.
- Better internal sales workflow.
- More accurate quote response emails.
- Client portal quote response display for portal-submitted quotes.
- More complete cost breakdown, including delivery, insurance, handling, taxes, freight, and any other applicable line items.
- Removing incorrect payment language from customer emails.
- Making sure every customer-facing quote response answers: “¿Por qué este es el precio?”

The goal is to make the whole quoting process easier for customers and much easier for the CRBOX sales team to process.

---

## Product Direction

The north star is:

- The public Cotizar page is the visual and UX reference.
- The client portal Cotizar experience should be the smarter version because CRBOX already knows the client.
- The internal sales tool should be the most complete operational workspace because sales needs to review, adjust, calculate, explain, and respond with precision.
- The customer response should be clear, transparent, and complete, whether delivered by email or rendered inside the client portal.

## Critical Principle: Do Not Fake Tariff, Tax, or Customs Data

Do not fake tariff, tax, or customs data.

Before implementing any “complete category list”, identify the official public source used in Costa Rica for tariff/customs/product tax categories. This may involve:

- Ministerio de Hacienda
- Dirección General de Aduanas
- TICA
- Sistema Arancelario Centroamericano / SAC
- Any official tariff table used for import duties and taxes

If the full official category list cannot be integrated immediately, implement the architecture so it can support it, and clearly document the limitation.

Do not pretend that the current partial list is complete.

The UI can support searchable intelligent category selection now, but the data source must be honest.

## Product Category / Tariff Data Upgrade

### Problem

The current product category selector is too limited.

CRBOX needs a much more complete category database that maps products to applicable customs/tax/arancel categories.

Users and sales agents should be able to search naturally instead of manually scrolling a small list.

### Requirement

Implement or prepare a complete searchable product category selector based on the official Costa Rican tariff/customs category source.

The category selector should work in:

- Public quote page.
- Client portal quote page.
- Internal sales calculator.

### Official Data Source Requirement

Before implementing the full list, identify the official public source for customs/tariff categories in Costa Rica.

Potential sources to investigate:

- Ministerio de Hacienda
- Dirección General de Aduanas
- TICA
- Sistema Arancelario Centroamericano / SAC
- Any official tariff table used for import duties and taxes

Once identified, document:

- Source name.
- Source URL or data origin.
- Data format.
- Whether it can be downloaded, scraped, imported manually, or maintained as a static JSON file.
- Fields available.
- How product category maps to tax/duty percentage.
- Whether there are special cases requiring permits or additional documentation.

### Category Selector UX

The selector should be searchable and intelligent.

It should allow the user to search by natural terms such as:

- zapatos
- laptop
- celular
- audífonos
- ropa
- repuestos de carro
- cosméticos
- juguetes
- suplementos
- libros
- perfumes
- herramientas
- muebles

The selector should show:

- Category name.
- Possible tariff/code if available.
- Tax/duty percentage if available.
- Short description if available.
- Warning badge if item may require permit/documentation.
- “No estoy seguro” option.

### No Match Behavior

If the user cannot find the right category, show:

“¿No encuentras la categoría exacta?”

Options:

- “No estoy seguro”
- “Solicitar revisión por ventas”
- “Describir producto manualmente”

If selected, the quote should still be submitted, but marked internally as:

categoryNeedsReview: true

Internal sales should see a clear badge:

“Categoría pendiente de revisión”

### Data Architecture

If no backend data source exists yet, create a reusable local/static data module such as:

- js/product-categories.js
- data/product-categories.json

The structure should support future import of the full official list.

Suggested structure:

{
  id,
  code,
  name,
  aliases,
  description,
  dutyRate,
  taxRate,
  totalEstimatedRate,
  requiresPermit,
  permitNotes,
  source,
  updatedAt
}

Do not hardcode category lists separately in multiple pages.

Use a shared module or shared endpoint where possible.

## Unit Conversion Logic

### Requirement

Use the same unit conversion logic across:

- Public quote page.
- Client portal quote page.
- Internal sales calculator.

The user and internal sales agent should be able to switch between:

Weight:

- lb
- kg

Dimensions:

- inches
- cm

The calculator must interpret conversions correctly and consistently.

### Behavior

If the user enters dimensions in inches, convert to cm internally if the calculation requires cm.

If the user enters weight in pounds, convert to kg internally if the calculation requires kg.

Show the selected unit clearly.

Avoid making the user do manual conversions.

### Internal Sales Tool

The internal calculator must support the same behavior.

Sales agents should be able to input the dimensions/weight in whatever unit they have from the customer or product page.

The system should convert and calculate normally.

Example:

- Weight: 10 lb
- Dimensions: 12 x 8 x 5 in

The system should calculate using the correct normalized values and show:

- Original input.
- Converted values.
- Applied billable weight if applicable.
- Whether real weight or volumetric weight was used.

## Delivery Logic

Delivery must be handled differently depending on the quote source and customer delivery method.

### Public Quote Page

If the user is not logged in, delivery may need to be selected manually or estimated.

Options can include:

- Retiro en sucursal
- Entrega a domicilio
- No estoy seguro

If the user selects delivery to home, include delivery cost if the current business rules support it.

If the user selects branch pickup, delivery cost should be $0 or “No aplica”.

### Client Portal Quote Page

Because the client is logged in, use known profile data when available:

- Preferred branch.
- Address.
- Delivery method.
- Sucursal.
- Locker number.
- Customer profile.

If the client’s selected delivery method is:

### Domicilio / Entrega a domicilio

Include delivery as a cost line item according to the current delivery pricing logic.

Show:

“Entrega a domicilio”

and the cost.

### Sucursal / Retiro en sucursal

Do not include a delivery charge.

Show:

“Retiro en sucursal: [Sucursal]”

Delivery line item:

“No aplica”

or

“$0.00”

The selected branch should be shown clearly.

### Unknown Delivery Method

If the delivery method cannot be determined:

- Ask user to choose.
- Or mark quote for sales review.

Do not silently apply a delivery charge.

## Quote Response Delivery: Public Email + Client Portal Response

The quote response experience depends on where the quote was submitted.

### Public Quote Submission

If the quote was submitted from the public Cotizar page, the customer receives the quote response by email.

The email must include the full customer-facing breakdown and disclosure:

- Product summary.
- Per-product calculation if there is more than one product.
- Total quote amount.
- Freight / shipping.
- Taxes / duties estimate.
- Handling / processing.
- Insurance.
- Delivery cost or “No aplica”.
- Payment methods: Sinpe Móvil and transferencia bancaria only.
- Clear estimate disclaimer.
- WhatsApp and email CTAs with proper spacing.

### Client Portal Quote Submission

If the quote was submitted from inside the client portal, the customer should receive the response in two places:

1. Email, using the improved quote response template.
2. Inside the client portal, attached to the submitted quote/request.

The portal response must not be a short generic status update.

It should show the full quote response with the same level of clarity as the email, but adapted to the portal UI.

Inside the client portal, show a quote response detail view with:

- Quote status.
- Quote ID.
- Date responded.
- Sales response / notes.
- Product list.
- Per-product breakdown.
- Total breakdown.
- Delivery method.
- Delivery cost or “No aplica”.
- Insurance.
- Taxes / duties estimate.
- Handling / processing.
- Freight / shipping.
- Final quoted total.
- Payment instructions.
- Support actions: WhatsApp / Email.
- Disclaimer explaining why the quote is an estimate and what can change.

The goal is for the logged-in customer to understand exactly why the price is what it is without needing to call or ask manually.

This only applies when the user quotes or orders more than one product.

Only show the consolidated calculation module when:

If the quote has one product, show the normal single-product breakdown instead.

This should reuse or align with the existing multi-product / consolidated calculation logic that already exists in the system.

Do not invent a separate calculation model.

The consolidated calculation UI should make clear:

- Each product has its own product value, category, weight, dimensions, taxes/duties, and subtotal.

  1. Per-product calculation

“Detalle por producto”

- Product name / description.
- Quantity.
- Category used for calculation.
- Product value.
- Real weight.
- Volumetric weight.
- Applied weight.
- Taxes/duties estimate.

## Full Disclosure Requirement

Every customer-facing quote response, whether email or portal, must answer:

“¿Por qué este es el precio?”

That means the user should clearly see:

- What product(s) were quoted.
- What category was used.
- What weight was applied.
- Whether volumetric or real weight was used.
- What taxes/duties were estimated.
- What handling or processing cost was applied.
- Whether insurance was included.
- Whether delivery was included.
- What the final total includes.

Avoid vague totals with no explanation.

Do not only show:

“Total: $X”

Always show the reasoning/breakdown behind the price.
