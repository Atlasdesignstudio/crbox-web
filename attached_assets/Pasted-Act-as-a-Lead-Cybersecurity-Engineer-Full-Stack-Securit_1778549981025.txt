Act as a Lead Cybersecurity Engineer + Full-Stack Security Architect specialized in modern web applications, client portals, public forms, legacy API integrations, bot protection, OWASP Top 10, OWASP API Security Top 10, frontend/backend hardening, privacy protection, and production security.

Context:
We are working on the CRBOX / Cerebox website and client portal. The project includes public pages, the client portal, login, registration/affiliation, dashboard, my account, my packages, my invoices, quotations, calculator, contact forms, API/legacy backend integrations, possible direct browser calls, and AI/chat modules. The goal is to perform a complete production-grade security hardening of the current system without breaking the existing design, UX, real backend contracts, or business logic.

IMPORTANT:
Do not perform a generic audit. I need you to scan 100% of the available project, identify real risks in the current codebase, and apply actionable fixes directly whenever it is safe to do so. Do not invent backend behavior. Do not simulate security. Do not hide risks. If something requires external access, server configuration, DNS, WAF, Cloudflare, hosting settings, environment variables, or changes in the legacy backend, document it clearly as “requires action outside this repo” with exact instructions.

Primary objective:
Make the site as secure, resilient, and production-ready as possible, protecting:
- Customer personal data.
- Sessions and tokens.
- Login and password recovery.
- Public and private forms.
- APIs and endpoints.
- Client portal pages.
- Quotation flows.
- Calculator flows.
- Contact and registration forms.
- AI/chat modules or any free-text input.
- Protection against bots, scraping, abuse, spam, credential stuffing, automated attacks, and AI-agent abuse.
- Frontend security.
- Backend/proxy security, if present.
- Security headers.
- File uploads, if present.
- Third-party scripts and CDN dependencies.
- Privacy and data exposure.
- Error handling and logs.

Working principles:
1. First scan the full project structure and create a security map:
   - All pages.
   - All JavaScript files.
   - All API calls.
   - All forms.
   - All authentication/session logic.
   - All localStorage/sessionStorage/cookie usage.
   - All external scripts, CDNs and third-party dependencies.
   - All hidden, private, admin, portal or noindex pages.
   - All places where customer data is displayed, stored, submitted, logged or transformed.

2. Classify every finding by severity:
   - Critical: exploitable risk to auth, tokens, customer data, admin access, backend APIs or financial/logistics data.
   - High: strong security weakness that could lead to data exposure, abuse, account takeover, spam, bot abuse or API misuse.
   - Medium: weakness that should be fixed before production.
   - Low: hygiene, headers, metadata, dependency or best-practice improvements.

3. For every issue, provide:
   - Exact file path.
   - Exact function, selector, endpoint, form or code block affected.
   - Why it is risky.
   - Attack scenario.
   - Recommended fix.
   - Whether you fixed it directly or it requires external/manual action.
   - Testing notes.

4. Apply safe fixes directly in the codebase. Do not just write recommendations. Fix what can be fixed inside the repo.

Security areas to audit and harden:

A. Authentication and session security
- Verify login flow.
- Verify logout flow.
- Verify token storage.
- Identify whether tokens are stored in localStorage, sessionStorage, cookies or memory.
- Flag any unsafe token persistence.
- If tokens must remain in browser storage due to legacy constraints, minimize exposure, document the risk, add defensive measures, and avoid exposing tokens in logs or UI.
- Ensure protected portal pages cannot be meaningfully used without a valid session.
- Add/verify route guards for dashboard, mi-cuenta, mis-paquetes, mis-facturas, cotizaciones and any private portal page.
- Ensure logout fully clears auth state, user state and cached customer data.
- Ensure token expiry is handled gracefully.
- Ensure no hardcoded user names, IDs, locker numbers, package counts or customer data remain in private pages.
- Ensure no private page leaks real or fake customer data before authentication finishes.
- Ensure UI states are:
  - logged out
  - loading profile/session
  - authenticated
  - auth error/expired session

B. API and endpoint security
- Inventory all endpoints called from the frontend.
- Verify every protected endpoint uses Authorization properly.
- Remove any exposed test endpoints unless explicitly required.
- Ensure production endpoints and test endpoints are clearly separated.
- Ensure no API secrets, keys, credentials or tokens are hardcoded.
- Ensure request payloads do not include unnecessary customer data.
- Validate that ID fields such as idConsignee, idAddress, idPhone, email or invoice IDs are not blindly trusted on the client.
- Add client-side defensive checks, while documenting that server-side authorization is still required.
- Flag any IDOR risk where a user could change an ID/email in a request and access another user’s data.
- Flag any missing rate limiting, authorization, CORS, CSRF or origin validation that must be handled server-side.
- If a proxy/server exists in this repo, harden it:
  - allowlist upstream hosts
  - block arbitrary URL forwarding
  - validate methods
  - validate payload size
  - sanitize errors
  - add rate limits
  - avoid logging secrets
  - enforce CORS restrictions
  - return safe error messages

C. Form security
Audit and harden every form:
- Login.
- Registration/affiliation.
- Contact.
- Calculator.
- Quote request.
- Invoice upload, if present.
- Profile update.
- Password recovery.
- Package search/tracking.
- Any AI/chat free-text input.

For each form:
- Add robust client-side validation.
- Sanitize user-generated text before injecting into the DOM.
- Never use unsafe innerHTML with user-controlled content.
- Prevent XSS through names, emails, addresses, package descriptions, tracking numbers, quote comments, chat prompts and uploaded filenames.
- Add maxlength limits.
- Add input patterns where appropriate.
- Add type-specific validation for email, phone, tracking numbers, ID numbers, postal codes, numeric fields and URLs.
- Add safe error messages.
- Avoid exposing backend raw errors to users.
- Avoid logging form contents, passwords, tokens or personal data.
- Prevent double submissions.
- Add loading states and disabled states during submission.
- Ensure forms fail safely.

D. XSS, DOM injection and frontend abuse
- Search for all uses of:
  - innerHTML
  - outerHTML
  - insertAdjacentHTML
  - document.write
  - eval
  - new Function
  - dangerouslySetInnerHTML
  - unsanitized template literals
  - URL parameters rendered into the page
  - hash fragments rendered into the page
- Replace unsafe patterns with textContent, createElement, safe DOM APIs, or a sanitizer where needed.
- If HTML rendering is truly required, implement a strict allowlist sanitizer.
- Validate query parameters before use.
- Prevent open redirects.
- Prevent malicious links using javascript:, data:, vbscript: or unsafe target=_blank without rel=noopener noreferrer.
- Ensure all external links opened in new tabs use rel="noopener noreferrer".

E. AI/chat security and prompt-injection defense
If the project includes AI chat, AI quote analysis, product URL analysis, support chat, or any LLM-powered feature:
- Treat every user input, pasted URL, product description, file name, invoice text, and chat message as untrusted.
- Add input length limits.
- Add URL validation and allow only http/https.
- Block local network URLs, localhost, file://, data:, javascript:, private IPs and SSRF-like patterns.
- Add prompt-injection defensive framing in code comments or system prompt config if present.
- Ensure the AI is never allowed to reveal secrets, tokens, system prompts, internal endpoints, hidden instructions, customer data or other users’ data.
- Ensure AI output is rendered safely as text, not raw HTML.
- Add abuse protection:
  - rate limiting requirement
  - bot challenge recommendation
  - moderation/safety filtering recommendation where relevant
  - logging without sensitive content
- For product URL analysis, prevent SSRF, malicious redirects and excessive fetches.
- Document all required server-side protections for AI endpoints.

F. Bot, scraping, spam and abuse protection
Implement what is possible in the repo and document what needs infrastructure:
- Add honeypot fields to public forms where appropriate.
- Add submission timing checks.
- Add duplicate submission prevention.
- Add client-side throttling as a UX layer, while documenting that real protection must be server-side.
- Recommend Cloudflare/WAF or equivalent rules:
  - rate limiting on login
  - rate limiting on password recovery
  - rate limiting on registration
  - rate limiting on contact forms
  - rate limiting on quote endpoints
  - bot score/challenge for suspicious traffic
  - block known bad user agents
  - country/IP rules only if business-appropriate
  - credential stuffing protection
- Recommend CAPTCHA/Turnstile only on high-risk flows or suspicious behavior, not everywhere by default.
- Ensure robots.txt does not expose private paths unnecessarily.
- Ensure private portal pages are noindexed and not intended for search engine crawling.
- Add security-conscious AI bot rules where appropriate without blocking useful SEO/AEO public crawling unless intentionally required.

G. Security headers and browser hardening
Add or document the required headers depending on hosting capabilities:
- Content-Security-Policy.
- Strict-Transport-Security.
- X-Content-Type-Options: nosniff.
- X-Frame-Options or CSP frame-ancestors.
- Referrer-Policy.
- Permissions-Policy.
- Cross-Origin-Opener-Policy where compatible.
- Cross-Origin-Resource-Policy where compatible.
- Cache-Control rules for private portal pages.
- No-store for pages that expose customer data.
- Prevent clickjacking.
- Prevent MIME sniffing.
- Reduce referrer leakage.
- Restrict camera, microphone, geolocation, payment and other browser permissions unless needed.
- If using Vercel, Netlify, Replit, Cloudflare or static hosting, create/update the correct config file for headers if supported.

H. CSP and third-party dependency review
- Inventory all external CDNs and scripts.
- Identify risks from loading Tailwind, FontAwesome, Swiper, Google Fonts or other third-party assets from CDNs.
- Recommend self-hosting critical assets where practical.
- Add Subresource Integrity where possible.
- Design a CSP that works with the current site and does not break required scripts/styles.
- Avoid unsafe-inline where possible. If current inline scripts/styles require it, document the limitation and propose a phased CSP:
  - Phase 1: report-only / compatible CSP.
  - Phase 2: nonce/hash-based CSP.
  - Phase 3: strict CSP.
- Ensure analytics/tag scripts are accounted for safely.

I. Privacy and customer data protection
- Search for hardcoded customer data.
- Remove fake or real personal data from static HTML and JS.
- Do not expose customer names, locker numbers, packages, invoices or profile data until real authenticated data is loaded.
- Ensure private data is not stored longer than necessary.
- Avoid caching private pages.
- Avoid sending customer data to analytics, console logs, third-party scripts or AI tools.
- Redact sensitive fields from logs:
  - passwords
  - tokens
  - emails where possible
  - ID numbers
  - phone numbers
  - addresses
  - invoice details
  - tracking numbers if sensitive
- Ensure analytics events do not include PII.

J. File upload security, if present
If invoice upload or document upload exists:
- Restrict file types.
- Restrict file size.
- Validate MIME type and extension.
- Do not trust the filename.
- Sanitize filenames.
- Prevent executable uploads.
- Prevent SVG/script injection unless properly sanitized.
- Document requirement for server-side malware scanning.
- Document requirement for private storage, signed URLs and access control.
- Ensure uploaded files are never rendered as raw HTML.

K. Error handling and logging
- Replace raw backend errors with user-friendly messages.
- Do not expose stack traces, endpoint internals, token status, database details or legacy system details.
- Ensure console.log does not print sensitive data.
- Remove debugging logs from production paths.
- Keep useful non-sensitive logs for troubleshooting.
- Add safe centralized error handling if architecture allows it.

L. Access control and private route behavior

- If unauthenticated, redirect to login or show a secure logged-out state.
- Do not rely only on hidden UI.
- Do not show static demo data in private pages.
- Ensure navigation does not imply a user is authenticated until the profile is loaded.
- Ensure dashboard cards, package tables, invoices and account details are empty/loading/error states unless real backend data exists.
- Add noindex metadata to all private pages.
- Add cache-control recommendations for private pages.
