This is not a redesign task. Preserve the current UI, layout, spacing, structure, visual hierarchy, cards, modals, badges, tabs, and interaction patterns as much as possible. If something is shown, it must be real, truthfully derived, or honestly unavailable / neutral / próximamente.

Expand the table/list rendering so the new portal does not silently regress against the legacy portal by throwing away real data already returned by the API.

- Estado → statusName (with badge/style driven by statusId)
- #MAWB → masterAirShipmentNumber
- #HAWB → airShipmentNumber
- Recibo → number
- Carrier → carrierName
- Tracking # → trackingNumber

- Envía / Tienda → shipperName
- Pieza → totalpieces
- Peso volumétrico → totalvolumetricweight
- Peso Kg → totalweight
- Notas → consigneeNotes
- Descripción → descripcion
- Acciones → keep exactly as today

Constraints:
- preserve the current Acciones column and existing handlers
- preserve horizontal scroll behavior on smaller screens
- no layout redesign
- loading / empty / error states must behave exactly as they do after the truthfulness pass

### 2) Mis Paquetes — grid view surfaces more real fields from the same array

### 3) Mis Paquetes — detail modal fully real

- Header:

- Shipping information block:
  - trackingNumber
  - receiveddatetime
  - carrierName

  - masterAirShipmentNumber
  - airShipmentNumber
- Dimensions / weight block:
  - totalpieces
  - totalweight

  - totalvolumetricweight

Requirements:
- the old fake “Costo Estimado” row must be removed if no real cost field exists
- any missing value must render as a neutral em dash, never a fake placeholder
- no hardcoded demo content

### 4) Mis Paquetes — tracking modal fully real and no demo timeline
When opened from a real package row, populate the tracking modal only from the selected real package:
- trackingNumber
- statusName
- carrierName
- receiveddatetime
- shipperName
- totalweight
- totalvolumetricweight
- consigneeNotes

Remove the static fake timeline/demo steps.

### 5) Mis Paquetes — re-audit package-adjacent informational blocks using the real fields
Evaluate these fields carefully:

- impresoFactura
- invoicesCount
- montofactura

- a fake consolidation savings percentage or fake progress bar is not defensible unless explicitly supported

Do not introduce any fabricated money amount, fake savings label, fake invoice progress, or fake readiness metric.

Do not invent invoice-line detail that is not present in the response.

### 7) Mis Facturas — truthfulness audit of summary cards and payment state
Re-audit the summary cards on Mis Facturas.

Rules:
- only keep metrics that are truthfully computable from the real bills response
- do not label invoices as Pagada or Pendiente unless there is a reliable explicit backend signal
- treat Factura.isInvoiced, createdDate, billedDate, PaymentDate, and PaymentRegistrationDate cautiously
- do not infer payment lifecycle semantics unless they are clearly defensible

If payment-state metrics are not defensible:
- replace with neutral / unavailable states
- do not fabricate any paid/unpaid counts

The payment-info block may remain as static business content, but it must not be implied to come from the backend.

### 8) Dashboard — re-evaluate derivations using both confirmed real shapes
Refine the dashboard where the newly confirmed package and bills shapes allow better derivations.

Rules:
- Paquetes en camino remains a real/derived metric using explicit statusId / statusName mapping
- Actividad reciente may be built from package data only if every displayed row has:
  - a real timestamp
  - a real status label
  - a defensible event interpretation
- skip rows that do not meet that standard
- if the resulting activity feed is empty, show Sin actividad reciente

For bills:
- only derive dashboard invoice-related metrics if supported by the bills response
- if not defensible, keep them neutral

No fake progress bars, no fake event feed, no invented invoice statuses.

### 9) Centralize field mapping
Introduce a single helper/accessor layer for packages and another for bills so all renderers read through consistent mappings.

For packages, every renderer (list row, grid card, detail modal, tracking modal, dashboard derivation) must read from one normalized mapping helper.

For bills, the table and any detail/expansion renderer must read through one normalized mapping helper.

This avoids duplicated field-name logic and silent inconsistencies.
