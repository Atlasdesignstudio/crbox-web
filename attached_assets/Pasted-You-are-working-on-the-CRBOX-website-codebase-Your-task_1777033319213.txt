## 1. Auth contract

Legacy portal uses bearer token auth.

### Login endpoint
POST `https://clients.crbox.cr/authtoken`

Content-Type:
`application/x-www-form-urlencoded`

Body:
- `grant_type=password`
- `username={email}`
- `password={password}`

Confirmed successful response shape:
```json
{
  "access_token": "...",
  "token_type": "bearer",
  "expires_in": 86399
}
```

The legacy app used `@websanova/vue-auth`, but in the new site you should implement auth cleanly for the current architecture.

### Auth header
All protected backend calls use:
`Authorization: Bearer {token}`

### Important auth notes
- token expiry must be tracked using `expires_in`
- login errors must be handled gracefully
- the legacy portal also stored the user email separately and used it to bootstrap `getuserinfo`
- in the new implementation, preserve that logic in a cleaner way

# Technical Requirements

## Auth storage
Implement a clean auth/session layer.

Recommended stored values:
- access token
- expires_at timestamp
- remembered email if needed for bootstrap
- remember-me preference

Support:
- session-only storage by default
- persistent storage when “Mantener sesión iniciada” is checked

On app load:
- if token exists and is still valid, user is authenticated
- if expired, clear auth state and treat user as logged out
- once authenticated, bootstrap profile with `getuserinfo(email)`

## API layer
Create a clean centralized API module, not scattered ad hoc fetch calls.

Suggested modules:
- `auth`
- `clientPortalApi`
- `profile mapper / serializer`

Support:
- login
- getUserInfo
- updateProfile
- getPackages
- getBills
- passwordRecovery
- registerUser

## Data normalization
Because backend shape is legacy/nested, create mappers between:
- backend response
- frontend portal state
- backend update/register payload

Do not couple raw response shape directly to every UI component.

## Empty states
Support clean empty states for:
- packages = `[]`
- bills = `[]`

## Error handling
Use user-friendly Spanish messages.
Do not expose raw implementation errors unless useful.

Handle:
- wrong credentials
- network issues
- expired token
- duplicate email
- duplicate ID
- profile update failure
- empty list states

## Loading states
Add real loading states for:
- login submit
- dashboard bootstrap
- profile save
- package search

- registration submit
- password recovery submit

## Protected routes
Client-area routes must require auth.

If unauthenticated:
- redirect to login page or the defined client login entry point
