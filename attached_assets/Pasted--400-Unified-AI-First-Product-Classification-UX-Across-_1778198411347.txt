#400 - Unified AI-First Product Classification UX Across All CRBOX Quote Touchpoints

What & Why

CRBOX already has the Product Brain and the AI-first classification pipeline foundation. However, the experience still feels dropdown-first or inconsistent in some places.

The goal of this task is to make the AI-first product/category experience visible, useful, and consistent across the entire CRBOX quote ecosystem.

This is NOT only for cotizar.html.

The same intelligence must work across:

- Public Cotizar page
- Public Calculadora page
- Product URL “Analizar”
- Client portal quote/cotización flow
- Internal sales/admin quote panel
- Admin calculator
- Quote response email
- Portal quote response
- Reminder email
- Floating CRBOX AI chat assistant

Every place where the user, client, or admin interacts with product category, product tax estimate, product URL analysis, product quote calculation, or product explanation must use the same Product Brain + Gemini classification logic.

---

## Core Objective

The user should not feel they are choosing from a technical category dropdown.

The user should feel that CRBOX understands what they want to bring.

A customer should be able to type or paste:

- “zapatos rosados Nike”
- “iPhone 15 Pro Max”
- “proteína whey”
- “power bank”
- “perfume Dior”
- “Stanley quencher”
- “air fryer”
- “drone DJI”
- “repuesto para Honda Civic”
- “maquillaje Sephora”
- Amazon / eBay / Walmart / Shein / Nike / Apple / Best Buy product URLs

And the system should:

1. Interpret the product naturally.
2. Assign the closest valid Product Brain category.
3. Auto-populate the compatible legacy category code.
4. Explain the estimated tax behavior in friendly Spanish.
5. Detect if the product may require manual review.
6. Detect if the product may be regulated, restricted, or forbidden.
7. Ask only for the missing fields.
8. Preserve the existing calculator and quote flow.

---

## Unified Classification Architecture

All touchpoints must use ONE shared pipeline:

Gemini interpretation  
→ Product Brain validation  
→ semantic approximation if exact product is missing  
→ normalized classification result  
→ existing calculator math  
→ friendly UX message  
→ admin metadata if needed

Use the shared module:

CRBOXProductClassifier

Do not create separate category logic per page.

Do not create separate keyword maps per page.

Do not create separate Gemini prompts per page unless they all call the same shared classification logic.

---

## Product Brain Role

Product Brain is the source of truth for:

- category
- legacy category code
- estimated tax range
- totalEstimatedRate
- manualReviewRequired
- automaticEstimateAllowed
- regulatedProduct
- restrictedProduct
- forbiddenProduct
- customerMessage
- adminNotes
- actionForCustomer
- actionForAdmin
- riskFlags

Gemini interprets.

Product Brain decides.

Calculator calculates.

---

## Gemini Role

Gemini should act as a Costa Rica courier/logistics product interpretation assistant.

Gemini should:

- interpret natural language product names
- interpret ecommerce product titles
- interpret product URL extraction results
- detect likely category
- detect missing fields
- detect risk patterns
- detect whether semantic approximation is needed

Gemini must NOT:

- invent tax rates
- invent Product Brain categories
- calculate final quote totals
- make final legal/customs decisions
- bypass Product Brain
- show raw reasoning to customers

If exact product does not exist in Product Brain, Gemini may map it semantically to the closest valid category.

Example:

“Stanley quencher pastel pink”
→ likely reusable tumbler / bottle
→ home/kitchen category
→ friendly explanation

Not:
“Product not found.”

---

## Critical UX Principle: No Hard Failure States

The system should not make normal uncertainty feel like failure.

Avoid user-facing messages like:

- “Producto no encontrado”
- “Categoría no encontrada”
- “AI failed”
- “No match”
- “Unknown category”
- “Error clasificando”
- “No se pudo analizar”

Use progressive, helpful language:

- “Detectamos una categoría probable para este producto.”
- “Parece pertenecer a Calzado.”
- “Solo falta completar el precio.”
- “CRBOX revisará algunos detalles para confirmar la cotización final.”
- “Este producto puede requerir revisión especial.”
- “No pudimos leer todos los datos del enlace, pero puedes completar lo que falta manualmente.”

The user should always feel guided forward.

Apply the same behavior to ALL of these:

### 1. Public Cotizar Page

File:
cotizar.html

Requirements:

- Product URL input uses the shared classifier after extraction.

### 3. Product URL “Analizar” Flow

Files:

cotizar.html
calculadora.html if applicable
portal quote flow if applicable

Requirements:

1. Product name/title
2. Product category
3. Price
4. Weight/dimensions only if available

If only partial information is extracted, preserve it.

Examples:

Only category detected:
“Detectamos una categoría probable. Solo falta completar el nombre y precio.”

Name + category detected:
“Detectamos el producto y su categoría. Solo falta confirmar el precio.”

Name + category + price detected:
“Producto detectado correctamente. Revisa la información antes de continuar.”

No full scrape:
“No pudimos leer todos los datos del enlace, pero puedes completar lo que falta manualmente.”

Do not make the experience depend on scraping everything.

### 9. Floating CRBOX AI Chat Assistant

Files:
chat assistant files
chat-calculator.js
chat-panel.js
crbox knowledge base files
Gemini/chat backend if applicable

Requirements:

The floating CRBOX AI assistant must also use the same Product Brain + CRBOXProductClassifier logic.

This is critical.

If a user asks in the floating chat:

- “¿Cuánto paga traer unos zapatos?”
- “Quiero traer un iPhone”
- “¿Puedo traer proteína?”
- “¿Puedo traer un power bank?”
- “¿Cuánto paga un perfume?”
- “¿Puedo traer un drone?”
- “¿Qué pasa si trae batería?”
- “¿Puedo traer un arma?”
- “Quiero traer esto de Amazon: [URL]”

The assistant must:

1. Route the product query through the shared classifier.
2. Use Product Brain category/rate/message.
3. Never answer from generic Gemini memory.
4. Never invent official rates.
5. Never present estimates as final customs liquidation.
6. Give friendly, useful explanations.
7. Ask for missing fields when needed.
8. Recommend manual CRBOX review for regulated/restricted products.
9. Direct the user to Cotizar/Calculadora when appropriate.

Examples:

User:
“¿Cuánto paga traer unos zapatos?”

Assistant:
“Los zapatos normalmente pertenecen a Calzado y suelen pagar aproximadamente 29% en impuestos. El monto final puede variar según material, valor y revisión aduanera. Puedes cotizarlo con el enlace o el nombre del producto.”

User:
“¿Puedo traer proteína whey?”

Assistant:
“La proteína whey puede requerir revisión sanitaria antes de importarse. CRBOX debe revisar el producto específico antes de confirmar si puede enviarse y cuánto pagaría.”

User:
“¿Puedo traer un power bank?”

Assistant:
“Un power bank contiene batería de litio, por lo que puede requerir revisión adicional de transporte. Compártenos el enlace o modelo para revisarlo.”

User:
“¿Puedo traer un arma?”

Assistant:
“Ese tipo de producto puede estar restringido o prohibido para courier. No recomendamos comprarlo antes de que CRBOX confirme si puede transportarse o importarse.”

The chat assistant should feel connected to the quote system, not like a separate generic bot.

## Manual Review Rules

If Product Brain says:

- automaticEstimateAllowed: false
- manualReviewRequired: true
- regulatedProduct: true
- restrictedProduct: true
- forbiddenProduct: true

Then:

- do not show final-looking automatic total
- show friendly review explanation
- allow user to submit for review if business rules allow
- mark product clearly for admin
- include actionForCustomer/actionForAdmin where useful
