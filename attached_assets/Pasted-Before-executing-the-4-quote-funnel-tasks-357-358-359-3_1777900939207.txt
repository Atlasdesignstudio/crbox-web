## 1. Script loading order for #357

Since tariff-adapter.js will now depend on product-categories.js, make sure product-categories.js is loaded before tariff-adapter.js on every page or admin view that uses the tariff adapter.

Expected order:

<script src="js/product-categories.js?v=..."></script>
<script src="js/tariff-adapter.js?v=..."></script>

Add defensive guards in tariff-adapter.js in case PRODUCT_CATEGORIES is unavailable, but do not rely on the guards as the normal loading path.

## 2. Persistence safeguard for #358

Before implementing quoteResponse persistence, audit how solicitud records are actually stored and updated.

Use the existing persistence mechanism only.

If there is already a safe backend update endpoint or storage pattern for saving structured data into the solicitud record, use it to persist quoteResponse.

If there is no safe way to persist quoteResponse yet, do not pretend that the portal response will be available. In that case:

- Implement the admin calculation UI and email response flow.
- Clearly document the missing persistence gap.
- Do not show fake portal response success.
- Keep #360 blocked or partially implemented until quoteResponse can be saved and read reliably.

This is important because #360 depends on quoteResponse being stored in a real place that solicitud.html / solicitud.js can read.

## 3. Live estimate language for #359

Any live estimate shown inside the client portal quote form must be labeled as an estimate / pre-cotización, not as the final quote.

Use language such as:

“Estimación preliminar”

or

“Pre-cotización de referencia”

Make it clear that final pricing is only confirmed after the CRBOX sales team reviews the request and responds.

Do not make the live estimate look like a final payable total.
