Problem summary:
When `USE_RDS_PACKAGES_FRONTEND=true`, `mis-paquetes.html` eventually calls `/api/portal/my-packages`.
That endpoint uses `_portal_auth()` to validate the user's Bearer token by making a server-side request from the Replit production VM to `clients.crbox.cr/getuserinfo`.
In production, that validation returned 401.
The same user session still worked through the direct browser legacy path, so the token was valid from the browser context.
Because the frontend treats RDS 401 as a real auth failure, it clears the session and redirects to login.
Rollback fixed the user-facing issue.

Goal:
Design the safest fix so RDS read endpoints do not wipe the user session when server-side legacy verification fails for infrastructure/network reasons.

Please produce a plan only.

Option A — Find/confirm a CRBOX API verification path that the production VM can call safely.

- Does it preserve user identity?
- Is it safe?
- What needs testing?

Option B — Change RDS auth failure classification.
- If `_portal_auth()` cannot verify the user because the server-side legacy call fails, return a non-auth fallback-safe status instead of 401.
- The frontend would fall back to legacy instead of clearing the session.
- But true invalid/expired sessions should still clear the session.

Option C — Use service-account verification.
