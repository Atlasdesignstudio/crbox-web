#385 - AI-First Product Classification Experience Using Product Brain + Gemini

## What & Why

We already built and expanded the CRBOX Product Brain.

Now we need to connect it into the real quotation experience.

The goal is to make the product/category experience AI-first, natural, consistent, and customer-friendly across every quote touchpoint.

Today, users often have to manually choose categories from dropdowns or rely on partial URL extraction. This feels technical, fragile, and inconsistent.

The new experience should feel like CRBOX understands what the customer wants to bring.

A user should be able to type or paste:

- “zapatos rosados Nike”
- “iPhone 15 Pro Max”
- “proteína whey”
- “power bank”
- “perfume Dior”
- “Stanley cup”
- “air fryer”
- “drone DJI”
- “repuesto para Honda Civic”
- “maquillaje Sephora”
- Amazon / eBay / Walmart / Shein / Nike / Apple / Best Buy product URLs

Then Gemini interprets the product, Product Brain validates/grounds the result, and the existing calculator code uses the compatible category/rate where appropriate.

This is NOT a calculator rewrite.

This is an intelligence + UX integration layer.

---

## Core Principle

Gemini is the primary interpretation layer.

Product Brain is the source of truth.

Calculator code remains the deterministic math engine.

Flow:

User enters product name or URL  
→ Gemini interprets product as a Costa Rica courier/logistics assistant  
→ Gemini returns likely product type, closest Product Brain category, missing fields, and risk signals  
→ System validates result against Product Brain  
→ If exact product does not exist, Gemini may semantically map it to the closest valid category  
→ Product Brain provides rate, category, warnings, customer message, admin notes, and review rules  
→ Existing calculator code uses the selected compatible legacy category where automatic estimate is allowed  
→ Admin can review/override if needed

Gemini must not invent rates.

Gemini must not invent final legal/customs conclusions.

Gemini must not calculate final shipping totals.

Gemini must not bypass Product Brain.

---

## Important UX Principle: No Hard Failure States

The Product Brain must not behave like a rigid exact-match database.

Gemini is allowed to semantically infer, approximate, and map products to the closest valid Product Brain category even when the exact product is not listed.

The goal is to preserve flow continuity.

Avoid user-facing messages like:

- “Product not found”
- “Category unknown”
- “AI failed”
- “Low confidence”
- “No match”
- “Error clasificando”
- “No se pudo analizar”

Instead, use helpful progressive guidance:

- “Detectamos una categoría probable para este producto.”
- “Parece pertenecer a Calzado.”
- “CRBOX revisará algunos detalles para confirmar la cotización final.”
- “Solo falta completar precio, peso o medidas.”
- “Este producto puede requerir revisión especial antes de confirmar.”

The user should never feel the system broke.

If the exact product is not found, the system should still try to:

1. infer the closest category,
2. explain the result positively,
3. ask only for the missing information,
4. allow submission for CRBOX review when appropriate.

Example:

User input:
“Stanley quencher pastel pink”

Even if “Stanley quencher” is not explicitly in Product Brain, Gemini should infer:
- reusable tumbler / bottle
- home/kitchen accessory
- likely category: hogar/decor/storage or kitchen accessory

Customer-facing result:
“Detectamos que este producto parece ser una botella térmica o accesorio de hogar. Esta categoría suele pagar aproximadamente 29% en impuestos. El monto final puede variar según material y revisión final.”

Not:
“Producto no encontrado.”

Example:

User input:
“zapatos barefoot minimalistas”

Even if “barefoot shoes” is not listed exactly, Gemini should infer:
- footwear
- category: calzado

Customer-facing result:
“Detectamos que este producto pertenece a Calzado. Esta categoría suele pagar aproximadamente 29% en impuestos.”

Example:

User input:
“DJI FPV battery”

Gemini should infer:
- drone accessory
- lithium battery
- special transport risk

Customer-facing result:
“Este producto puede requerir revisión especial porque parece incluir batería de litio y estar relacionado con drones. CRBOX debe validarlo antes de confirmar si puede transportarse y cuánto pagaría.”

Create one shared AI-first product classification pipeline used by all product/category/quote touchpoints.

All touchpoints must behave consistently:

- Public cotizar.html

- Product URL “Analizar” flow
- Client portal Cotizaciones/Solicitudes flow

## Gemini Role

Gemini should behave like a Costa Rica courier/logistics product interpretation assistant.

Gemini should analyze:

- natural language product input
- long ecommerce product titles
- product URL extraction results
- partially extracted data
- user notes
- visible product name/description
- possible category
- possible risk factors
- whether the product is likely regulated
- whether the product may include batteries, liquids, food, medicine, chemicals, telecom, drones, weapons, etc.

Gemini should return:

- detectedProductName
- likelyProductType
- bestMatchingProductBrainCategoryId
- closestCategoryReason
- possibleRiskFlags
- missingFields
- shortFriendlyReason
- whether manual review is recommended
- whether semantic approximation was used

Gemini instruction:

“Return only an existing Product Brain category ID. Do not invent a new category. Do not invent tax rates. Do not calculate shipping. Do not make final legal/customs decisions. If the exact product is not listed, semantically map it to the closest valid Product Brain category. If the category is still unclear or risky, return unknown_manual_review. Always preserve a helpful customer experience.”

---

## Product Brain Validation

After Gemini returns a category ID:

1. Check that the category ID exists in Product Brain.
2. If valid, load the full Product Brain category record.
3. Use Product Brain data for:
   - legacyCode
   - estimatedDAI
   - vatRate
   - law6946Rate
   - totalEstimatedRate
   - estimatedRange
   - customerMessage
   - adminNotes
   - actionForCustomer
   - actionForAdmin
   - manualReviewRequired
   - automaticEstimateAllowed
   - regulatedProduct
   - restrictedProduct
   - forbiddenProduct
   - riskFlags
4. If Gemini category does not exist, fall back to semantic local matching.
5. If still unclear, use unknown_manual_review.
6. If Gemini is unavailable, use ProductBrain.classifyLocal() as fallback.
7. If local fallback gives approximate match, return semantic_approximation instead of hard failure.

Important:
Gemini interprets.
Product Brain decides.
Calculator calculates.

## Semantic Approximation Rules

The system should not require exact product rows.

If the exact product does not exist in PRODUCT_PRODUCTS, Gemini can map it to the closest category based on:

- product function
- material
- use case
- ecommerce category
- product title
- brand context
- similar products in Product Brain
- risk pattern
- tax behavior group

Examples:

“Stanley quencher” → bottle/tumbler → home_decor_storage or kitchen accessory  
“barefoot shoes” → footwear_complete  
“Owala bottle” → home_decor_storage  
“DJI propellers” → drones_rc_review or cameras/photo depending context, but flag drone  
“MagSafe battery pack” → lithium_batteries_powerbanks  
“Lululemon leggings” → clothing_general  
“Dyson Airwrap” → beauty_devices  
“Kindle Paperwhite” → tablets_ereaders  
“Meta Quest strap” → gaming_consoles_electronics / gaming accessory  
“Hydro Flask” → home_decor_storage  
“Theragun” → beauty_devices or medical/fitness depending context, flag battery_possible  
“Car oxygen sensor” → automotive_parts_review  
“Dog vitamins” → pet_food_supplements_review  
“Retinol serum” → beauty_skincare_personal_care, flag sanitary_possible  
“Creatine gummies” → supplements_vitamins_nutrition  
“Paint marker” → office_stationery_art or chemicals_aerosols_adhesives depending liquid/flammable context

If approximation is used, customer copy should say:

“Detectamos una categoría probable…”

or

“Parece pertenecer a…”

not:

“Producto no encontrado.”

## URL Analyze Experience

This is one of the most important UX improvements.

When the user pastes a product URL and clicks “Analizar”:

1. Existing URL extraction attempts to get product name, price, category, weight, dimensions.
2. Preserve whatever data is successfully extracted.
3. Pass extracted product title/name/description into CRBOXProductClassifier.
4. Gemini/Product Brain classifies the product.
5. UI shows partial success when some information is missing.
6. Ask the user only for missing fields.

Examples:

If AI gets product name and category but not price:
“Encontramos el producto y su categoría. Solo falta confirmar el precio para completar la cotización.”

If AI gets product name but not dimensions:
“Detectamos el producto correctamente. El peso y las medidas no aparecen en el enlace, por lo que el equipo CRBOX podrá revisarlas al preparar la cotización.”

If AI cannot scrape much but can infer category:
“Pudimos identificar el tipo de producto. Revisa o completa los datos faltantes antes de enviar.”

If request truly fails:
“No pudimos leer todos los datos del enlace, pero puedes completar la información manualmente.”

Avoid:

- “AI failed”
- “Extraction failed”
- “No category found”
- “Error”

## Natural Manual Input Experience

When a user manually types product name:

- Do not force dropdown-first selection.
- User can type naturally.
- Use local lightweight suggestions while typing if helpful.
- Use Gemini-first analysis when user clicks “Analizar producto” or pauses intentionally.
- Debounce to avoid excessive API calls.
- Category selector can remain as override, but should not be the primary UX.

Recommended flow:

1. User types product name.
2. Local Product Brain shows lightweight suggestions.
3. User clicks “Analizar producto” or system analyzes after a clear pause.
4. Gemini interprets.
5. Product Brain validates.
6. Category field is auto-populated.
7. Friendly explanation appears.
8. User can keep or change manually.
9. If user changes manually, re-render Product Brain message for selected category.

## Manual Review Rules

If Product Brain says:

- automaticEstimateAllowed: false
- manualReviewRequired: true
- regulatedProduct: true
- restrictedProduct: true
- forbiddenProduct: true

Then:

- Do not show a final-looking automatic total.
- Show friendly warning.

- Mark product clearly for admin.

- Do not encourage purchase.

## Error / Partial Extraction UX

The experience should never feel broken when AI cannot extract everything.

Use progressive completion language.

Examples:

“Este enlace no muestra peso o medidas claras. Puedes enviar la solicitud y CRBOX lo revisará.”

“Detectamos una categoría probable, pero CRBOX puede revisarla antes de confirmar.”

“No pudimos analizar el producto automáticamente en este momento. Puedes continuar completando los datos manualmente.”
