Act as a Senior Web Performance Engineer, Core Web Vitals specialist, and production frontend optimization lead for CRBOX.

Your job is to run a complete performance, speed, Core Web Vitals, and production loading audit across the entire CRBOX public website and client portal. This is not a passive audit. It is an actionable performance hardening sprint.

Scan the entire project, identify every issue affecting loading speed, rendering, mobile performance, perceived performance, layout stability, asset delivery, script execution, CSS loading, third-party loading, API-driven loading states, and production stability. Then fix everything that can be fixed safely inside the codebase.

Context:
CRBOX includes a public website and client portal, including pages such as:
- Homepage
- Servicios
- Cómo funciona
- Tarifas
- Calculadora
- Contacto
- Afíliate / registration
- Login
- Dashboard
- Mi cuenta
- Mis paquetes
- Mis facturas
- Cotizaciones / quote flow
- Any other public, portal, private or support page found in the repo

Primary goal:
Make the website and portal feel fast, stable, lightweight, responsive, and production-grade on both mobile and desktop without changing visible design, layout, copy, brand voice, business logic, analytics intent, SEO intent, authentication behavior, or backend contracts.

Important:
Do not redesign anything.
Do not rewrite copy.
Do not remove functionality.
Do not fake performance by hiding important content.
Do not break analytics, SEO, schema, auth, portal data, forms or API behavior.
Do not change the visual design unless the change is technically invisible or necessary to prevent layout instability.
Do not make speculative optimizations that risk breaking the site.
Preserve the current visual experience.

Performance principles:
1. Improve real user experience, not just synthetic scores.
2. Prioritize mobile performance.
3. Prioritize above-the-fold rendering.
4. Prevent layout shift.
5. Reduce unnecessary blocking resources.
6. Keep third-party scripts under control.
7. Preserve crawlability and analytics.
8. Preserve truthful loading, empty and error states.
9. Make improvements production-safe and reversible.
10. Document anything that requires hosting/CDN/backend changes.

PHASE 2 — Core Web Vitals audit

Audit and optimize for:

Largest Contentful Paint:
- Identify likely LCP element per page.
- Check if the LCP image/text is delayed by CSS, JS, lazy loading or render-blocking resources.
- Ensure hero images are not lazy-loaded.
- Add fetchpriority="high" only to the true primary hero image when appropriate.
- Add width and height to hero images.
- Add decoding strategy where useful.
- Ensure critical visible content is available in HTML and not unnecessarily delayed.

Cumulative Layout Shift:
- Add explicit width and height to images.
- Reserve space for banners, cards, forms, embedded widgets, and dynamic portal modules.
- Prevent late-loading fonts from causing major shifts.
- Prevent dynamically injected content from moving the layout unexpectedly.
- Avoid loading states that collapse and then expand.
- Ensure portal cards, tables and empty states have stable dimensions where appropriate.

Interaction to Next Paint:
- Detect heavy click handlers.
- Detect unnecessary JavaScript on pages that do not need it.
- Defer non-critical scripts where safe.
- Avoid repeated DOM scans if unnecessary.
- Avoid expensive scroll listeners without throttling/debouncing.
- Avoid large synchronous work on user interaction.
- Ensure calculators and filters respond quickly.
- Ensure portal filters and search do not freeze the UI.

First Contentful Paint:
- Reduce render-blocking resources where safe.
- Add or correct preconnects for critical third-party origins.
- Avoid loading unnecessary page-specific JS on all pages if possible.
- Avoid redundant CSS or duplicate external libraries.
- Ensure fonts are loaded efficiently.

Time to Interactive / Total Blocking Time:
- Add defer to non-critical scripts where safe.
- Avoid duplicate script execution.
- Remove unused or obsolete JS when clearly safe.
- Split or conditionally initialize page-specific behavior where practical.
- Ensure third-party scripts do not block core UI.

PHASE 3 — Image and media optimization

Audit every image:
- Missing width/height.
- Missing loading strategy.
- Missing decoding.
- Overly large image dimensions.
- Incorrect image format.
- Broken image path.
- Duplicate image usage.
- Image used above the fold but lazy-loaded.
- Image below the fold not lazy-loaded.
- Decorative images that should not affect accessibility/SEO.
- Portal images or icons that shift layout.

Fix directly where safe:
- Add width and height attributes based on actual or intended display dimensions.
- Add loading="lazy" to below-the-fold images.
- Remove lazy loading from above-the-fold hero images.
- Add fetchpriority="high" only to one true primary hero image per page where appropriate.
- Add decoding="async" where useful.
- Add stable containers around images if needed to prevent CLS.
- Flag images that should be compressed, converted or resized externally.

Do not:
- Replace images with different creative.
- Crop or visually alter images.
- Change visible layout.
- Add fake placeholders that change the design.

PHASE 4 — CSS performance audit

Audit:
- Number of CSS files per page.
- Duplicate CSS rules.
- Inline styles that can cause maintenance/performance issues.
- Render-blocking CSS.
- Unused page-specific CSS loaded globally.
- Heavy animations.
- Expensive box shadows or filters used excessively.
- Mobile CSS overrides that cause layout instability.
- Repeated CSS variables or inconsistent tokens.
- CSS that causes horizontal overflow on mobile.
- CSS that causes large layout recalculations.

Fix directly where safe:
- Remove clearly duplicated CSS only if it does not change visuals.
- Consolidate repeated rules only when low risk.
- Ensure responsive CSS prevents overflow.
- Ensure animations respect performance and do not trigger excessive layout.
- Prefer transform/opacity for animations where applicable.
- Preserve the current visual design.

PHASE 5 — JavaScript performance audit

Audit:
- All scripts loaded on every page.
- Scripts that should be page-specific.
- Blocking scripts without defer.
- Duplicate event listeners.
- DOMContentLoaded handlers that initialize features not present on the page.
- Expensive scroll listeners.
- Expensive resize listeners.
- Repeated querySelectorAll calls inside scroll/resize/click loops.
- Large calculator logic duplication.
- Dashboard animation overhead.
- Third-party library usage.
- Console logs left in production.
- Broken JS causing runtime errors.
- JS that blocks forms or navigation.

Fix directly where safe:
- Add defer to local scripts where safe.
- Guard page-specific code so it only runs when relevant elements exist.
- Throttle/debounce scroll and resize handlers when needed.
- Remove production console logs unless intentionally debug-only.
- Prevent duplicate event binding.
- Make calculators and filters efficient.
- Ensure runtime errors do not break the rest of the page.
- Preserve all behavior.

Do not:
- Rewrite the app architecture unless necessary.
- Remove features.
- Break analytics events.
- Break portal interactions.
- Break mobile menus.
