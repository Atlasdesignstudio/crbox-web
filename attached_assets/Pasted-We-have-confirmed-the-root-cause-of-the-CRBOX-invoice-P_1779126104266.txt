We have confirmed the root cause of the CRBOX invoice/PDF issue. Please solve this carefully without breaking the new website, the legacy admin, or any existing data.

Context:
The new CRBOX website is now live on crbox.cr. However, the legacy admin still uses invoice/PDF URLs under:

https://crbox.cr/wp-content/uploads/...

Those files do not exist on the new server, so invoice previews/downloads in the legacy admin are failing with 404.

Confirmed evidence:
The broken live URL:

https://crbox.cr/wp-content/uploads/2026/05/FacturaCompra-WR606103.pdf

currently returns 404 from the new server:

server: SimpleHTTP/0.6 Python/3.11.14
via: google

But when we test the old WordPress server directly using:

curl -I --resolve crbox.cr:443:98.90.3.205 https://crbox.cr/wp-content/uploads/2026/05/FacturaCompra-WR606103.pdf

it returns:

HTTP/1.1 200 OK
Server: Apache
Content-Type: application/pdf

So the files are not lost. They are still being served correctly from the old WordPress server at:

98.90.3.205

The problem is that crbox.cr now points to the new site, so legacy WordPress upload URLs are hitting the wrong server.

Goal:
Keep the new CRBOX website live for normal public routes, but restore legacy invoice/PDF access and invoice upload behavior permanently.

Required fix:
Implement a safe legacy passthrough/proxy in the new CRBOX server/app so that these paths are forwarded to the old WordPress server at 98.90.3.205:

1. /wp-content/uploads/*
2. /wp-json/crbox/v1/saveBill

- crbox.cr/
- crbox.cr/servicios
- crbox.cr/tarifas
- crbox.cr/calculadora

But:

- crbox.cr/wp-content/uploads/*

And:

- crbox.cr/wp-json/crbox/v1/saveBill
