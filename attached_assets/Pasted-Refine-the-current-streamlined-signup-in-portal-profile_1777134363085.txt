Refine the current streamlined signup + in-portal profile completion plan into a more complete, product-quality implementation plan and execution scope.

The current direction is correct, but it still leans too much toward payload simplification and not enough toward the full user experience, activation logic, navigation model, and real backend validation.

I want this refined so the final outcome is not just “remove fields from afiliate and show a banner later,” but a genuinely better account creation + setup + activation experience that feels intentional, user-friendly, and fully grounded in what the current backend can and cannot do.

Core objective

We want to redesign the account creation flow so CRBOX stops asking for a long, dense, operationally-heavy form before the user can even create an account.

Instead, we want a model closer to:

* fast, low-friction initial signup
* real account creation in the backend
* automatic login into the client portal
* clear in-portal setup / activation flow
* completion of the remaining operational data at the right moment and in the right context

But this must be done in a way that:

* respects the real backend contracts
* does not create broken or low-quality accounts
* does not leave users confused after entry
* does not simply relocate the same dense form into the portal without rethinking the experience

⸻

What this refinement must explicitly cover

1. This is not only a form-reduction task

Expand the plan so it clearly treats this as a combined:

* signup UX task
* onboarding UX task
* activation-state product task
* navigation/deep-linking task
* backend-truth validation task

Do not frame the solution as just:

* fewer fields on afiliate.html
* more fields later in mi-cuenta.html

That is not enough.

The refined plan must ensure the end-to-end experience feels:

* easier
* lighter
* better designed
* more guided
* more coherent
* more conversion-friendly
* more operationally safe

⸻

2. Validate first what the real backend truly supports

Before finalizing the new signup flow, explicitly validate which minimal payload the real backend accepts stably across the full lifecycle, not just on the initial registration response.

The refined plan must require validation of:

* registration succeeds
* auto-login succeeds
* getuserinfo returns a coherent user record
* the portal loads correctly
* later profile completion / editing still works
* no downstream page or module breaks because of missing deferred fields
* the created account is operationally valid, not just technically inserted

Make it explicit that:

A field is not considered “safely deferrable” just because the registration endpoint accepts an empty value once.
It is only safely deferrable if the entire account lifecycle remains stable and coherent afterward.

This matters especially for:

* identification fields
* address structures
* sucursal / delivery preference
* phone structure
* profile editing
* any portal logic that assumes those fields exist later

If the backend does not safely support the desired lightweight signup, the plan must adapt the minimum initial step accordingly instead of forcing placeholders.

⸻

3. Define the account states clearly

The refined plan must explicitly define the product states that result from this new flow.

At minimum, define:

A. Account created

The user exists and can authenticate.

B. Profile incomplete

The user is inside the portal, but still lacks some data needed for full CRBOX operational use.

C. Operationally activated account

The user has completed the minimum required profile data for CRBOX to fully support the intended use.

Do not leave this vague.

The refined plan must answer questions like:

* what exact fields are required to count as “activated”?
* does activation require identification?
* does it require a branch/delivery preference?
* does it require a full address only for domicilio?
* what can the user still do before activation?
* what should be blocked or deferred until activation?
* how is this state communicated?

This needs to be a real product/state model, not just “show setup if something is missing.”

⸻

4. Design the in-portal setup experience intentionally

The portal setup/completion experience must be designed as a guided onboarding flow, not as “the same dense old form moved somewhere else.”

The refined plan must explicitly require the agent to design the best possible user experience for the setup stage inside the client portal.

That includes:

* how the setup is introduced
* where it appears
* how strongly it is surfaced
* how the user navigates through it
* how dense information is grouped
* how missing information is prioritized
* how explanations are written
* how the user knows what is left
* how the user reaches the correct tab or section
* how this works on both mobile and desktop

It is fine if this uses Mi Cuenta as the actual completion surface, but the experience should feel:

* progressive
* guided
* contextual
* easier than the current signup form
* intentionally designed for completion

Not just “go fill this big form.”

⸻

5. Do not rely only on a dashboard banner

The current idea of using a dashboard setup banner is good, but it is not enough by itself.

Refine the plan so the incomplete-account experience is more robust than a single dismissible card.

The plan should explicitly define:

* whether the dashboard card exists
* whether Mi Cuenta reflects incomplete state clearly
* whether deep links open the correct tab
* whether the user continues seeing setup prompts across sessions if required data is still missing
* whether dismissal only hides the card visually or actually changes state
* whether key actions are gated until activation

The user should never feel:

* lost
* unsure what is missing
* unsure why something is blocked
* like the product forgot they still need setup

⸻

6. Make the initial afiliarse page a true high-conversion entry point

The refined plan must explicitly cover the UX and design of the new lightweight signup page, not just its data payload.

Treat afiliate.html as a serious conversion surface.

The refined plan should require the agent to improve:

* clarity of the initial account choice
* simplicity of the personal signup path
* honesty of the business account path
* trust and scannability of the screen
* order and grouping of fields
* mobile-first usability
* CTA clarity
* error and success states
* what happens after successful signup
* what expectations are set about “you can complete the rest inside your portal”

The result should feel clearly easier than today.

⸻

7. Be careful with domicilio and address handling

The current idea of simplifying delivery preference and potentially using light address handling is good directionally, but it needs more caution.

Refine the plan so that:

* address collection for domicilio is only simplified in ways that remain structurally compatible with the backend and later editing flows
* if a minimal domicilio payload produces unstable, poor-quality, or misleading records, then a better solution is to defer the full structured address completion to the portal
* the plan should prefer clean deferred completion over messy placeholder address bootstrapping

Do not oversimplify address capture if it creates technical debt or bad operational records.

⸻

8. Define behavior if the user does not complete setup

The refined plan must explicitly include the UX/product rules for non-completion.

For example:

* what happens if the user signs up and enters the portal but ignores setup?
* what happens across later sessions?
* which portal modules remain usable?
* which actions, if any, require activation first?
* how do we remind them without making the experience annoying?
* how does the system avoid confusion while still nudging completion?

This should be intentional and documented.

⸻

9. Keep Google Sign-In out for now, but make the architecture future-compatible

Do not implement Google Sign-In in this task.

But the refined plan should explicitly say that the new model should be structured in a way that is compatible with a future auth pattern where:

* signup/login can happen quickly
* operational profile completion happens afterward inside the portal

In other words:
don’t build Google now, but do build the product logic in a way that would support that later.

⸻

10. Improve the final deliverable/report expectations

Refine the final report requirements so the agent must explain not only what changed, but the product logic behind it.

The final deliverable/report should include:

1. which fields remained in initial signup and why
2. which fields were moved into deferred in-portal completion and why
3. which fields were initially considered deferrable but were kept upfront due to backend or lifecycle risk
4. which values/placeholders were tested and whether they were actually safe
5. what defines:
    * account created
    * profile incomplete
    * operationally activated
6. how the in-portal setup experience is surfaced and guided
7. how the product behaves when the user postpones setup
8. which actions remain available vs. gated before activation
9. any real backend limitations that prevented a more aggressive simplification
10. confirmation that the final experience was considered from both UX and backend-truth perspectives, not just payload reduction
