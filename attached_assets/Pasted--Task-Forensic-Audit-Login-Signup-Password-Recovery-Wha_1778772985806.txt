# Task — Forensic Audit: Login, Signup & Password Recovery

What & Why
Before moving toward a safer production cutover/domain transition, we need a forensic audit of the current authentication and account-creation flows:

- Login
- Signup / afiliación / registration
- Password recovery / forgot password

These flows are more sensitive than read-only portal modules because they involve authentication, tokens, account creation, passwords, validation, and possible writes to the legacy CRBOX system.

This task is audit-only. Do not implement changes yet.

Goal
Map exactly how login, signup, and password recovery currently work across the new portal, legacy CRBOX APIs, browser storage/session handling, redirects, payloads, validation, and security boundaries.

The result should be a clear technical and product recommendation for what should remain legacy, what can be safely modernized, and what requires further discovery before replacing anything.

Important
Do NOT:
- Change login behavior.
- Change signup behavior.
- Change password recovery behavior.
- Change password reset/change behavior.
- Change auth tokens.
- Change cookies/session handling.
- Modify production secrets.
- Modify AWS/RDS settings.
- Modify DB schema.
- Enable any RDS frontend flags.
- Run writes except safe existing flow observation if absolutely necessary and explicitly non-destructive.
- Create test users unless explicitly approved.
- Change customer data.
- Log passwords, tokens, full payloads, raw ID numbers, phone numbers, or sensitive data.

This is a forensic audit only.

Relevant flows to audit

1. Login
Audit:
- Login page file(s).
- Form fields.
- Validation rules.
- Endpoint called.
- HTTP method.
- Payload format.
- Required headers.
- Response shape.
- Where access_token is stored.
- Token expiration handling.
- Logout behavior.
- Redirect after login.
- Error handling.
- Loading states.
- Whether login relies on:
  - legacy CRBOX `/authtoken`
  - local proxy
  - Replit server
  - browser direct call
- Whether any token or password is logged.
- Whether token is stored in localStorage/sessionStorage/cookie.
- Whether current storage is acceptable or should eventually move to HTTP-only cookie.
- Whether login works consistently across:
  - public site
  - portal pages
  - direct deep link to portal page
  - expired session
  - logged-out user

Known current login endpoint to verify:
`POST https://clients.crbox.cr/authtoken`
Expected payload style to verify:
`application/x-www-form-urlencoded`
with:
- grant_type=password
- username
- password

3. Password Recovery / Forgot Password
Audit:
- Password recovery UI.
- Form fields.
- Endpoint called.
- HTTP method.
- Payload/query format.
- Whether email is sent by legacy backend.
- Response shape.
- Success/error messaging.
- Whether response leaks account existence.
- Rate limiting / abuse protection if visible.
- Whether request requires auth or is public.
- Whether email is sent securely.
- Whether UI message is generic enough:
  “If this email exists, instructions have been sent”
  vs exposing whether account exists.
- Whether current behavior is acceptable or needs hardening.

Known endpoint to verify:
`GET https://clients.crbox.cr/api/crboxwebapi/getuserpasswordrecovery/{email}`

4. Cross-flow session behavior
Audit:
- What happens after login.
- What happens after logout.
- What happens when token expires.
- What happens if user opens:
  - dashboard.html
  - mis-paquetes.html
  - mis-facturas.html
  - mi-cuenta.html
  directly while logged out.
- Whether redirect target is preserved.
- Whether browser back button exposes protected content after logout.
- Whether bfcache restore is handled.
- Whether multiple tabs behave correctly.
- Whether `auth.js`, `portal-api.js`, `nav-auth.js`, `mobile-drawer.js` interact safely.

5. Security review
Check:
- Passwords are never placed in URL.
- Tokens are never logged.
- Tokens are not exposed in query params.
- Passwords are only sent over HTTPS.
- Error messages do not leak sensitive info unnecessarily.
- No credentials are hardcoded in frontend.
- No production secrets are exposed.
- No fake auth state.
- No fake success state.
- No console logs with tokens/passwords/PII.
- No raw access_token in docs/reports.
- No raw password in screenshots/logs.

Also classify current token storage:
- localStorage
- sessionStorage
- cookie
- memory
- other

Then recommend:
- Acceptable for current phase
- Should move to HTTP-only cookie later
- Must fix before production cutover

6. Data/write risk classification
For each flow, classify risk:

Login:
- Auth-sensitive, but no account write except token issuance.

Signup:
- High-risk write flow.
- Creates or modifies account data.
- Requires stronger validation and backend clarity.

Password recovery:
- Security-sensitive public flow.
- Abuse/rate-limit risk.

Password change:
- If included in mi-cuenta, classify separately.
- Do not test live password change unless explicitly approved.
