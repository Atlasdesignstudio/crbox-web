# Fix Unit Handling Across Quote + Calculator Flows (Public + Portal)

## What & Why
There is a real UX and data-quality problem across CRBOX’s product-entry flows:

- Public Cotizar page
- Public Calculadora page (inside the Agregar artículo modal)
- Client portal → Cotizaciones → Nueva cotización form

Right now, these flows assume a rigid input model for:
- dimensions in cm
- weight in kg

But in real usage, many product pages — especially Amazon / US retailers — show:
- dimensions in inches
- weight in pounds

This causes two major issues:

1. Users either guess the conversion or enter values in the wrong unit.
2. Volumetric weight / estimate context becomes inaccurate because the system interprets inch values as cm, or pounds as kg.

This needs to be solved properly and consistently across all relevant product-entry surfaces, not as a one-off patch in a single page.

The goal is to let users enter dimensions and weight in the units they actually have, while keeping the backend and business logic safe and normalized.

---

## Core principle
Do not create separate pricing logic by unit.  
Do not fork the backend contract.  
Do not send ambiguous mixed-unit data downstream.

Instead:

- allow flexible units in the UI
- normalize all physical values into one canonical internal model
- keep all downstream calculations and backend payloads based on the normalized values

### Canonical internal target
Unless inspection proves otherwise, the system should normalize to:
- cm for dimensions
- kg for weight

But do not assume blindly — verify the current logic first.

## Main product requirement
Implement one intelligent unit-handling approach that works across:

- Public Cotizar
- Public Calculadora → Agregar artículo modal
- Portal → Cotizaciones → Nueva cotización

The user must be able to enter:
- dimensions in cm or in
- weight in kg or lb

The system must:
- normalize those values safely
- preserve correct calculations
- avoid breaking any backend contract
- send downstream data in the correct canonical format

---

## Done looks like
- Public Cotizar accepts dimensions in cm or in
- Public Cotizar accepts weight in kg or lb
- Public Calculadora article modal accepts dimensions in cm or in
- Public Calculadora article modal accepts weight in kg or lb
- Portal Cotizaciones “Nueva cotización” form accepts dimensions in cm or in
- Portal Cotizaciones form accepts weight in kg or lb
- All three surfaces use the same normalization logic
- Volumetric weight is always calculated from normalized values, not raw display values
- The backend still receives clean, expected, canonical physical data
- No duplicate estimate logic is created
- No unit ambiguity remains in summaries or payloads
- The UI remains elegant, simple, and low-friction

## UX direction
This should feel natural and lightweight.

### Dimensions
For the three dimension inputs:
- Length / Width / Height
- or Largo / Ancho / Alto depending on the page language

Add a shared unit selector for dimensions:
- cm
- in

### Important
Do not create one selector per field.  
Use one shared dimensions unit control for the group.

### Weight
Add a separate unit selector for weight:
- kg
- lb

### Best UX direction
Each form / modal should have:
- one dimension unit selector
- one weight unit selector

This keeps the UI clean and understandable.

---

## Conversion rules
Use reliable standard conversions:

- 1 in = 2.54 cm
- 1 lb = 0.45359237 kg

Normalize into canonical internal values:
- length_cm
- width_cm
- height_cm
- weight_kg

### Optional but recommended
If useful for UX/debugging, preserve raw input context too:
- length_input
- width_input
- height_input
- weight_input
- dimension_unit
- weight_unit

But the business logic / calculation / backend payload must always have normalized values too.

---

## Volumetric weight
This is one of the key reasons for the fix.

Review how volumetric weight is currently computed in:
- Public Cotizar
- Public Calculadora
- Portal Cotizaciones

Then make sure:
- volumetric weight is always derived from normalized cm values
- never from raw UI values unless they are already cm
- never from unit-ambiguous inputs

### Example of the current failure
If a user enters:
- 3 × 2 × 1
but those values came from a product page in inches,
and the system interprets them as cm,
then the volumetric weight becomes wrong.

That exact class of problem must be eliminated.

## Shared implementation requirement
Do not implement three separate solutions.

Create a shared conversion / normalization utility that all relevant flows can use.

### Preferred direction
A shared helper such as:
- convertInToCm()
- convertLbToKg()
- normalizePhysicalInputs()

or equivalent clean abstraction.

The important part is:
- one source of truth for conversion
- one source of truth for normalization
- reused across public and portal product-entry flows

- non-numeric values

Preserve current optional/required rules where appropriate, but make unit handling robust.

## Email / request summary behavior
Wherever CRBOX receives or displays quote/article physical data, the output should be truthful.

Recommended behavior:
- preserve the original entered unit for human readability if useful
- always compute from normalized canonical values
- never silently display unitless numbers that can be misread

If a summary includes dimensions, consider a structure like:
- original entered dimensions + unit
- normalized dimensions in cm only where operationally helpful
- volumetric weight in the canonical result unit

Use judgment to avoid clutter, but remove ambiguity.

---

## Backend safety requirement
This change must be implemented without breaking the backend.

That means:
- inspect what the backend currently expects
- keep canonical data flowing in the expected format
- do not introduce ambiguous mixed-unit payloads
- do not change pricing/business logic contracts unnecessarily

If raw unit metadata is added, it should be additive and safe — not a replacement for normalized fields.
