PHASE 4 — Structured data implementation
Add JSON-LD schema.org markup where valid and truthful.

Use only data that exists in the repo or is visible on the page. Do not invent.

Recommended schemas:
1. Organization schema
   - Name: CRBOX
   - URL: https://crbox.cr
   - Logo if available
   - SameAs only if real social links are present
   - ContactPoint only if real contact information is present

2. WebSite schema
   - Site name
   - URL
   - PotentialAction SearchAction only if the site has a real search function. Do not fake it.

3. WebPage schema
   - For every main public page
   - Include name, description, url, inLanguage, isPartOf

4. BreadcrumbList schema
   - Where breadcrumbs exist or can be inferred safely from visible navigation
   - Must match visible hierarchy

5. Service schema
   - For servicios.html and service-specific sections only when content supports it
   - Use conservative, truthful descriptions
   - Do not invent price ranges or ratings

6. FAQPage schema
   - Only if visible FAQ content exists on the page
   - The JSON-LD questions/answers must exactly correspond to visible FAQ content
   - Do not create invisible FAQ content

7. LocalBusiness or PostalService / ShippingService-related schema
   - Only if business address/contact data is present and truthful
   - Do not invent opening hours, geo coordinates, ratings, branches or service areas

8. SoftwareApplication or WebApplication schema
   - Only if the calculator is clearly a web tool and schema is appropriate
   - Keep it conservative

Implementation rules:
- Prefer one clean JSON-LD block per page or a small number of organized blocks.
- Avoid duplicate conflicting schema.
- Ensure schema validates with Google Rich Results Test and Schema.org validator.
- Ensure canonical URLs in schema match canonical tags.
- Ensure image URLs are absolute when required.
- Ensure all structured data is truthful and supported by page content.

PHASE 5 — robots.txt, sitemap.xml and crawl files
Create or update:

1. robots.txt
- Allow crawling of public pages and assets needed for rendering.
- Do not block CSS, JS, images required for rendering public pages.
- Disallow private/admin/client portal routes where appropriate.
- Include sitemap reference.
- Do not expose sensitive internal paths unnecessarily.
- Be careful: robots.txt is public, so do not list secret endpoints or sensitive admin paths in a way that reveals them unnecessarily.

2. sitemap.xml
- Include only canonical public URLs.
- Exclude private pages, login if it should not be indexed, dashboard, account pages, packages, invoices, admin, API routes.
- Include lastmod only if reliable.
- Use correct absolute URLs.
- Use consistent priority/changefreq only if helpful; avoid fake precision.

3. llms.txt
- Add a concise public map for AI systems.
- Include:
  - Site name
  - Public purpose
  - Important public pages
  - What each page is for
  - Public contact/support page
  - Explicit note that private client portal/account data is not public and should not be indexed or used for answers
- Do not include private implementation details.
- Do not include API endpoints.
- Do not include customer data.
- Do not include secrets.

4. Optional: sitemap-index.xml
- Only if site scale requires it.

PHASE 7 — Accessibility as crawlability
Make non-visual semantic improvements that help both users and machines:
- Ensure images have meaningful alt text or empty alt="" if decorative.

- Ensure main content has <main>.

PHASE 8 — Performance and rendering for crawlers
Without changing design:
- Add width/height to images if missing.
- Add loading="lazy" for below-the-fold images.
- Avoid lazy-loading above-the-fold hero images.
- Add fetchpriority="high" to the most important hero image only if appropriate.
- Add decoding="async" where useful.
- Add preconnect for required third-party origins.
- Remove duplicate preconnects.
- Avoid unnecessary prefetching of private pages from public pages.
- Ensure CSS/JS required for rendering is not blocked.
- Check large assets and flag optimization opportunities.
- Do not visually alter layout.

PHASE 9 — Analytics and SEO safety
Check analytics/tag scripts:
- Ensure analytics scripts do not block crawl rendering.
- Ensure no PII is sent in SEO metadata, URLs, events or structured data.
- Ensure no private user data appears in public source.
- Ensure tracking parameters do not create duplicate canonical issues.
- Ensure canonical tags remove query duplication concerns.

PHASE 10 — Backend / hosting / deployment SEO configuration
If the repo includes hosting config, server config, Vercel, Netlify, Replit, Express, Nginx, Apache or similar:
- Add/verify headers for:
  - text/html
  - sitemap.xml
  - robots.txt
  - llms.txt
  - assets
- Add Cache-Control:
  - Public assets: long cache with immutable where fingerprinted
  - HTML: appropriate short cache/revalidation
  - Private pages: no-store if served
- Ensure redirects are canonical:
  - HTTP → HTTPS
  - non-canonical host → canonical host
  - duplicate routes → canonical route
- Ensure 404 returns proper status if backend exists.
- Ensure sitemap and robots are served with correct content types.
- If hosting configuration cannot be changed in repo, document exact required external settings.
