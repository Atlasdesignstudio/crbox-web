# Task — Prepare RDS Frontend Logging Instrumentation Plan

What & Why
The RDS observability plan concluded:

- Level 1: production shadow compare is ready with current prefix logs.
- Level 2: production frontend flag enablement requires small logging improvements before any user-facing RDS flag is enabled.

This task prepares the exact implementation plan for those logging improvements, but does not implement them yet.

Goal
Create a focused implementation plan for the five required improvements:

1. Add `duration_ms` timing to the three RDS frontend handlers.
2. Emit structured success log lines.
3. Add `fallback_reason` propagation in `portal-api.js`.
4. Sanitize exception logging.
5. Define monitoring coverage for the first 2 hours after flag enablement.

Deliverable
Create:

`docs/rds-frontend-logging-instrumentation-plan.md`

Important
Do NOT:
- Implement code.
- Enable RDS flags.
- Query production.
- Modify credentials.
- Modify AWS/RDS.
- Change endpoint behavior.
- Log raw PII.
- Log tokens.
- Log full payloads.
- Touch production env vars.

This is a planning document only.

Use as input:
- `docs/rds-observability-fallback-plan.md`
- `server.py`
- `js/portal-api.js`
- `docs/rds-production-readiness-plan.md`
- current RDS endpoint implementation

Scope
Cover exactly these user-facing RDS paths:

Backend:
- `_handle_portal_my_packages`

- `_handle_portal_profile_rds`

## 4. Backend instrumentation plan

For each backend handler:

- packages
- invoices
- profile

Document:

### Success path
Should emit:
- event: `rds_request_success`
- module
- endpoint
- status_code
- duration_ms
- flag_state
- db_guard_result
- count
- source: `rds`

### Feature disabled path
Should emit, if useful:
- event: `rds_feature_disabled`
- module
- endpoint
- status_code: 503
- flag_state: false
- error_type: `feature_disabled`

### Auth error path
Should emit:
- event: `rds_auth_error`
- status_code: 401 or 403
- error_type: `auth_error`
- fallback_triggered: false

### Wrong database path
Should emit:
- event: `wrong_database_guard_triggered`
- status_code: 503
- db_guard_result: `fail` or `env_not_set`
- error_type: `wrong_database`

Do not log the actual database names.

### RDS error path
Should emit:
- event: `rds_request_error`
- status_code
- error_type
- sanitized exception class only
- duration_ms

Do not log raw exception text unless sanitized.

## 5. Frontend fallback instrumentation plan

- classify non-auth failures into an error category
- attach a safe `rdsErrorCategory` or `fallbackReason`

- do not send tracking numbers
- do not send invoice numbers
- do not expose token
- auth errors should not fallback

Document whether fallback logging should be:
- browser console only
- server-side only
- GA4/GTM event
- not analytics, logs only

Recommend the safest option.
