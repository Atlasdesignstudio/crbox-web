We need a full product, UX, systems, and implementation architecture proposal for the future of the CRBOX quote / assisted-purchase experience, both:

1. outside the client portal (public-facing)
2. inside the client portal (logged-in customer experience)

This must be treated as a serious product-design, systems-thinking, workflow-design, and implementation-planning task — not a UI brainstorm.

I want the proposal to be:
- deeply detailed
- grounded in the current CRBOX reality
- realistic with today’s technology
- realistic with today’s CRBOX system
- explicit about what is feasible now vs later
- explicit about what AI can truly do today vs what is still speculative
- opinionated and concrete
- useful for actual implementation planning, not just strategy slides

If something is not feasible today, do not hand-wave it.  
Say clearly:
- why it is not feasible
- what dependency is missing
- what the honest fallback should be
- what would be needed later to unlock it

This should not be fantasy.  
It must be operationally usable.

--------------------------------------------------
CORE CONTEXT
--------------------------------------------------

We are evolving CRBOX from:
- a shipping calculator
- a quote handoff by email / WhatsApp
- and a client portal for package/account management

into a smarter system where users can:
- request quotes
- request assisted purchases
- paste product links
- get AI-assisted extraction and prefill
- send quote requests to CRBOX
- and, over time, manage those quote requests more fully inside the portal

Important current reality:
- there is not yet a new internal CRBOX back-office tool for managing quote workflows
- today, CRBOX sales can still receive requests by email and respond manually
- later, this may evolve into a richer internal + portal workflow
- the proposal must work now, not only in a future ideal state

Also important:
- the public quote experience and the portal quote experience should not be identical
- they should share logic where appropriate, but they have different jobs and different UX goals

--------------------------------------------------
HIGH-LEVEL PRODUCT GOAL
--------------------------------------------------

The user should no longer feel like they are just “using a calculator.”

The experience should feel like:
- I want to buy or bring this product
- I paste a link
- the system helps me structure the request
- I understand an estimate
- I send it to CRBOX
- CRBOX reviews it
- and, over time, I can track it inside my portal

So this is not just a calculator redesign.
It is the design of a quote intake / assisted purchase request system.

--------------------------------------------------
NON-NEGOTIABLE PRINCIPLES
--------------------------------------------------

1. Grounded AI only
AI should help:
- extract
- prefill
- guide
- classify
- ask for missing fields
- reduce friction

AI should not:
- invent exact costs
- invent guaranteed weights or dimensions
- invent official tariff truth
- silently confirm uncertain data as fact
- make the user think the system knows more than it really does

2. Public and portal must be intentionally different
Public = lightweight, conversion-oriented, easier first-touch quote flow  
Portal = recurring, trackable, richer, more native, more stateful

3. Design for now and later
The proposal must support:
- what CRBOX can actually operate today
- while preparing cleanly for future workflow expansion

4. No fake automation
If a flow still depends on email/manual review, say so.
If a flow can be partially automated, define the boundary clearly.

5. Respect real operational friction
Today CRBOX may still quote manually, review manually, and answer outside the portal.
The system should reduce friction without pretending the whole pipeline is already automated.

6. Do not confuse estimate with commitment
A quote request, a suggested estimate, an assisted purchase request, and a confirmed purchase may be related, but they are not automatically the same thing. Define those boundaries clearly.

PART 1 — PUBLIC QUOTE EXPERIENCE (OUTSIDE THE CLIENT PORTAL)
--------------------------------------------------

Design the ideal public-facing quote experience.

This experience is for:
- any visitor
- non-logged-in users
- people exploring CRBOX for the first time
- people who want a quick estimate
- people who may later convert into account creation

The public flow should preserve the current value of the quote tool, but make it much smarter and easier.

### What I want for the public flow
The user should be able to:
- quote manually
- or paste a product link
- trigger AI-assisted extraction / prefill
- review and correct extracted values
- calculate a quote
- send it to CRBOX
- and then be invited to create an account while momentum is high

### Public AI behavior
If a user pastes a URL:
- the system should be able to use Gemini (or equivalent AI layer) to try to extract:
  - product title
  - merchant/domain
  - visible price
  - category suggestion
  - possible dimensions if explicitly shown
  - possible weight if explicitly shown
  - model/spec info if clearly present
- the system should mark fields as:
  - extracted
  - inferred
  - missing
  - needs user confirmation
- the system should clearly ask for whatever is still missing
- the user must remain able to edit all relevant fields before calculating

### Public conversion requirement
After a quote is valuable enough, the experience should invite the user to create an account.

I want you to design:
- where that invitation appears
- how it should be framed
- how to preserve momentum
- how to pass along already-entered user information (name/email if available)
- how to route them into signup without making them feel like they are starting over

### Public-to-portal continuity
I want you to explicitly define:
- whether a public quote can later become associated to a newly created account
- how that linking would work safely
- whether draft/public quote data can survive signup
- whether public quotes should be imported into portal history after signup
- what identity matching strategy is safest

### Public experience output needed
Describe:
- the ideal page structure
- the entry points
- the AI-assisted flow
- the manual fallback flow
- the send-to-CRBOX handoff
- the signup conversion moment
- the mobile experience
- the empty states
- the failure states
- and the best UX copy direction

--------------------------------------------------

This section must be very realistic.

The AI layer should be able to:

### I want you to define:

- how unsupported or failed extraction should degrade gracefully
- how the manual fallback works
- how AI-assisted prefill and manual editing coexist
- how you would structure the prompts / JSON outputs conceptually
- what is feasible today with Gemini and what is not
- which types of sites/pages are likely to work better or worse
- what additional infrastructure is needed beyond “an API key”

### Human review boundary
This is critical. Define explicitly:
- what fields can be safely extracted and used as provisional values
- what fields should always be user-confirmed
- what fields should always remain human-reviewed by CRBOX before any final quote/purchase action
- what should never be auto-confirmed by AI

### Important
You must clearly state:
- what today’s AI can realistically do well
- what it can do only partially
- and what should remain manual
